# 3.1  数据链路层的功能
## 3.1.0 前言
### 一、数据链路层的主要任务
- 数据链路层的主要任务是——实现帧在一段链路上或一个网络中进行传输
### 二、数据链路层的三个基本问题
 - 数据链路层协议有多种，但有三个基本问题则是共同的，即
	 - 封装成帧
	 - 透明传输
	 - 差错检测
### 三、数据链路层的两种信道
#### 1. 点对点信道
- 使用**一对一**的通信方式。
	- <font color="#ff0000">PPP 协议</font>则是目前使用<font color="#ff0000">最广泛</font>的点对点协议。
#### 2. 广播信道
- 这种信道上连接的主机很多，使用**一对多**的广播通信方式。
	- 采用**共享广播信道**
		- <font color="#ff0000">有线</font>局域网普遍使用<font color="#ff0000"> CSMA/CD</font> 协议，
		  
		- <font color="#ff0000">无线</font>局域网则使用<font color="#ff0000"> CSMA/CA</font> 协议。
## 3.1.1 数据链路层所处的地位
### 一、图例解释
>下面使用两台主机通过互联网进行通信的例子来了解数据链路层所处的地位
#### 1. 连接
![[Pasted image 20240424225612.png]]
- 局域网 1 中的主机 H 1 经过路由器 R 1
  
- 广域网及路由器 R 2 连接到局域网 2 中的主机 H 2。

#### 2. 从协议的层次上看数据流动
![[Pasted image 20240424225705.png]]
- 主机 H 1 和 H 2 都有**完整的五层协议栈**，而路由器在转发分组时**仅使用协议栈的下三层**。
- 当主机 H 1 向 H 2 发送数据时
	- 数据进入路由器后要先从物理层上到网络层，（物理层——>网络层）
	- 在转发表中找到下一跳的地址后，再下到物理层转发出去。
	- 因此，数据从主机 H 1 送到主机 H 2 需要**在路径中的各结点的协议栈向上和向下流动多次**。

#### 3. 总结
> [!note] 注意
> - 当我们学习数据链路层时，通常可以只关心协议栈中<font color="#ff0000">水平方向</font>的各数据链路层。

![[Pasted image 20240424230025.png]]
- 于是，当主机 H 1 向主机 H 2 发送数据时, 可以想象**数据**就是在各相关设备的数据链路层之间**沿水平方向传送**的。
	- 即通过以下这样的链路: 
		- H 1 的链路层→R 1 的链路层→R 2 的链路层→H 2 的链路层，
		- 其中**三段不同的数据链路**可能**采用不同的数据链路层协议**。
### 二、点对点通信的基本概念
#### 1. 链路
- 指从**一个结点到相邻结点的一段物理线路**。
- 当进行数据通信时，两台计算机之间的通信路径往往要经过许多段这样的链路。
- 可见链路只是一条路径的组成部分。
#### 2. 数据链路
- 数据链路的组成
	- 链路+实现通信协议的软硬件
		- 当在一条链路上传送数据时，除了需要链路本身，还需要一些必要的通信协议来控制这些数据的传输，把实现这些协议的硬件和软件加到链路上，就构成了数据链路。
- 有时也把上面所说的链路称为物理链路，而把数据链路称为逻辑链路。
	- 链路——>物理链路
	- 数据链路——>逻辑链路
#### 3. 帧
- 数据链路层**对等实体**之间进行<font color="#ff0000">逻辑通信的协议数据单元</font>。
- 数据链路层
	- 把网络层下交的数据构成帧发送到链路上
		- 网络层数据——>传给链路
	- 把接收到的帧中的数据取出并上交给网络层
		- 收到的数据——>网络层
## 3.1.2 为网络层提供服务
### 一、无确认的无连接服务
- 源主机发送帧时不需要先建立链路连接，
- 目的主机收到帧时不需要发回确认。
- 数据传输的可靠性由高层负责
- 适用于误码率较低的信道，如**以太网**。

|       |         |
| ----- | ------- |
| 源主机   | 无需建立链路  |
| 目的主机  | 收到帧无需确认 |
| 可靠性   | 高层负责    |
| 适用误码率 | 较低      |
| 应用    | 以太网     |

### 二、有确认的无连接服务
- 源主机发送帧时不需先建立链路连接，
- 但目的主机收到帧时必须发回确认。
- 源主机在所规定的时间内未收到确定信号时，就重传丢失的帧，以提高传输的可靠性。
	- 超时重传
- 该服务适用于误码率较高的信道，如无线通信。

|       |         |
| ----- | ------- |
| 源主机   | 无需建立链路  |
| 目的主机  | 收到帧必须确认 |
| 可靠性   | 源主机负责   |
| 适用误码率 | 较高      |
| 应用    | 无线通信    |
- 无连接服务都不要求源主机建立链路连接
### 三、有确认的面向连接服务
- 帧传输过程分为三个阶段: 
	- 建立链路
	- 传输
	- 释放链路
	  
- 目的主机对收到的每一个帧都要返回确认。
	- <font color="#ff0000">逐帧返回确认</font>
- 该服务适用于**可靠性要求较高**的场合。

|      |          |
| ---- | -------- |
| 源主机  | 需要建立链路   |
| 目的主机 | 逐帧确认     |
| 可靠性  | 高        |
| 误码率  | 可靠性较高的场合 |
| 应用   | /        |

> [!attention] 注意
> - 有连接就一定要有确认，即**不存在无确认**的**面向连接**的服务，
## 3.1.3 链路管理
#### 一、基本概念
##### 1. 链路管理的概念
- 数据链路层<font color="#ff0000">连接的建立、维持和释放过程</font>称为链路管理
	- 主要用于**面向连接**的服务
##### 2. 结点如何通信
- 链路两端的结点如何通信
	- 必须首先确认对方已处于就绪状态，
		- 并交换一些必要的信息以对帧序号初始化
	- 然后才能建立连接
		- 在传输过程中要能维持连接
	- 而在传输完毕后要释放该连接

总结：
- 先确认双方准备就绪，同时交换必要信息初始化
- 建立连接以及维持连接
- 传输完毕断开连接
## 3.1.4 封装为帧与透明传输
#### 一、封装成帧 [[#3.2 组帧]]
##### 1. 封装成帧的概念
- 封装成帧是指——在一段**数据的前后**分别<font color="#ff0000">添加首部和尾部</font>，构成帧
##### 2. 帧、帧长、帧定界、帧同步
###### 1. 帧
- 帧是数据链路层的数据传送单元。
###### 2. 帧长
- 帧长等于帧的数据部分长度加上首部和尾部的长度。
###### 3. 帧定界
- 首部和尾部中含有很多控制信息，它们的**一个重要作用**是<font color="#ff0000">确定帧的界限</font>，即帧定界。
###### 4. 帧同步
- 接收方能从接收到的二进制比特流中<font color="#ff0000">区分出帧的起始与终止</font>，即帧同步。
##### 3. 如何提高帧的传输效率、最大传送单元
- 为了提高帧的传输效率，
	- 应当使帧的<font color="#ff0000">数据部分的长度尽可能地大</font>于**首部和尾部的长度**，
		- 但随着帧长的增加，传输差错发生的概率也随之提高，
		- 发生差错时重传的代价也越大，
		- 因此每种链路层协议都规定了帧的数据部分的长度上限，即最大传送单元。
- 最大传送单元。
	- 帧的数据部分的长度上限
##### 4. HDLC 帧格式以及如何帧同步
###### 1. HDLC 帧格式
![[Pasted image 20240425000743.png]]
###### 2. 帧同步
- 如在 HDLC 协议中，
	- 用标识位 F (01111110) 来标识帧的开始和结束。
	- 在通信过程中，
		- 检测到帧标识位下即认为其是帧的开始，
		- 然后一旦检测到帧标识位 F 即表示帧的结束[]()
#### 二、透明传输
##### 1. 透明传输的概念
- 若在数据中恰好出现与**帧定界符相同的比特组合**
	-  会<font color="#ff0000">误认为“传输结束”而丢弃后面的数据</font>
- 则要采取**有效的措施**来解决这个问题，即透明传输。

##### 2. 确切说明
- 更确切地说，
	- 透明传输是指
		- <span style="background:#d3f8b6">不论所传的数据是什么样的比特组合</span>，都能够**按原样无差错**地在这个数据链路上传输。
		- 不论什么样的数据都能在数据链路上**完整无差错传输**
## 3.1.5 流量控制
#### 一、流量控制的产生背景
- <font color="#00b050">因为链路两端结点的工作速率和缓存空间存在差异</font>，
	- 所以发送方的发送能力可能**大于**接收方的接收能力，
- 此时若**不适当限制**<font color="#ff0000">发送方的发送速率</font>，
	- 前面来不及接收的帧将被后面不断发送来的帧“淹没”，**造成帧的丢失**而出错。
	  
- 出错原因
	- 发送方的数据流太大，超过接收方的承受范围
	- 接收方被迫丢弃来不及收到的帧

- 流量控制
	- 流量控制**实际上**就是**限制发送方的发送速率**，使之不超过接收方的接收能力。
		- 这个过程需通过**某种反馈机制**，
			- 使发送方知道在什么情况下可以接着发送下一帧，
			- 而在什么情况下必须暂停发送，以等待收到某种反馈信息后继续发送。
			- <span style="background:#d3f8b6">知道什么时候发，什么时候不发，以及什么时候继续发</span>
#### 二、在OSI 体系与TCP/IP 的区别

|           | OSI                |     | TCP/IP           |
| --------- | ------------------ | --- | ---------------- |
| 流量控制位于    | 数据链路层              |     | 传输层              |
| 控制什么的数据流量 | 控制的是相邻节点之间数据链路上的流量 |     | 控制的是从源端到目的端之间的流量 |

## 3.4.6 差错检测
#### 一、两种错误
- 因为信道噪声等原因，帧在传输过程中可能会出现错误，
	- 这些错误分为
		- 位错
			- <font color="#ff0000">帧中**某些位**</font>出现差错，
			- 通常采用<span style="background:#d3f8b6">循环冗余检验 (CRC) </span>来发现位错
		- 帧错
			- <font color="#ff0000">帧丢失、帧重复或帧失序</font>等错误，
			- 它们都属于<span style="background:#affad1">传输差错</span>。
#### 二、过去的OSI观点
- 过去 OSI 的观点是: 
	- <font color="#ff0000">必须让数据链路层向上提供可靠传输</font>。
	  
	- 因此在 **CRC 检错的基础上**，**增加**了<font color="#ff0000">帧编号</font>、<font color="#ff0000">确认和重传机制</font>。
		- 收到正确的帧就要向发送方发送确认。
			- <span style="background:#affad1">接收方成功接受就返回确认</span>
		- 发送方在一定期限内若未收到对方的确认，就认为出现了差错，因此进行重传，直到收到确认为止。
			- 即<span style="background:#affad1">超时重传直至确认</span>
#### 三、现在的观点
##### 1. 在通信质量较差的无线传输中
   - 数据链路层依然使用**确认和重传机制**，向上提供**可靠**的传输服务。
     
##### 2. 在通信质量良好的有线链路
- 数据链路层**已不再使用确认和重传机制**，即**不要求**向上提供**可靠**传输的服务，
- 而**仅需进行 CRC 检错**，
	- 目的是将有差错的帧丢弃，保证上交的帧都是正确的，
- 而对出错的帧的重传任务则由高层协议 (如传输层 TCP 协议) 完成。

# 3.2 组帧
## 3.2.0 前言
### 一、组帧——封装成帧
- 发送方依据**一定的规则**将网络层递交的分组封装成帧 (也称组帧)。
	- 分组————>帧
### 二、组帧的原因
- 数据链路层之所以要将**比特组合**成**以帧为单位传输**，
	- 是为了在出错时只重发出错的帧，
	- 而不必重发全部数据，从而提高效率。
	- 分段打包，出错重传错的包
  
- 组帧主要解决
	- 帧定界
	- 帧同步
	- 透明传输等问题。
### 三、实现组帧的四种方式
- 字符计数法
- 字节填充法
- 零比特填充法
- 违规编码法
### 四、注意
- 组帧时**既要加首部，又要加尾部**。
	- 区分帧的边界
	- 原因是
		- 在网络中信息是以帧为最小单位进行传输的，
			- 所以**接收方要正确地接收帧**，就**必须清楚**<span style="background:#affad1">该帧在一串比特流中从哪里开始到哪里结束</span>
			-  (因为接收方收到的是一串比特流，没有首部和尾部是不能正确区分帧的)。
			  
		- 而**分组 (即 IP 数据报)** 仅是包含在帧中的数据部分 (后面将详细讲解)，所以**不需要加尾部来定界**。
## 3.2.1 字符计数法
### 1. 定义
- 字符计数法是指
	- 在**帧首部**使用一个**计数字段**来<font color="#ff0000">记录该帧所含的字节数</font>
		-  (包括计数字段自身所占用的 1 个字节)
### 2. 如何实现
![[Pasted image 20240425104750.png]]
- 当接收方读出帧首部的字节计数值时，就知道后面跟随的字节数，从而确定帧结束位置。
	- <span style="background:rgba(240, 200, 0, 0.2)">告诉有多长，知道结尾在哪</span>
- 因为**帧与帧**之间是**连续传输**的，所以**也能确定下一帧的开始位置**。
### 3. 最大问题
- 这种方法最大的问题在于
	- 若**计数字段出错**，即**失去帧边界划分的依据**
	- 则接收方就无法判断所传输帧的结束位和下一帧的开始位，收发双方将失去同步，造成灾难性后果。
## 3.2.2 字节填充法
### 1. 定义
- 字节填充法使用特定字节来定界一帧的开始与结束
### 2. 特定字节
![[Pasted image 20240425133659.png|475]]
- 在图 3.6 的例子中，
	- 控制字符 SOH 
		- 放在帧的最前面，表示帧的开始，
	- 控制字符 EOT 
		- 表示帧的结束。
	- 转义字符ESC
		- <span style="background:rgba(240, 200, 0, 0.2)">为了使信息位中出现的特殊字符<font color="#ff0000">不被误判为帧</font>的首尾定界符</span>，可在特殊字符之前填充一个转义字符 ESC 来加以区分以实现数据的透明传输。
			-  (注意，转义字符是 ASCI 码中的控制字符，是一个字符，而非“E”“S”“℃”三个字符的组合)，
		- 接收方收到转义字符后，就知道其后面紧跟的是数据信息，而**不是控制信息**
### 3. 如何实现
- 在图 3.6 (a) 所示的字符帧中,
	-  帧的数据段中出现 EOT 或 SOH 字符, 发送方在每个 EOT 或 SOH 字符前再插入一个 ESC 字符——见图 3.6 (b)
	- 接收方收到数据后会自己删除这个插入的 ESC 字符, 结果仍得到原来的数据——见图 3.6 (c)。
	  
	- 这也正是**字符填充法**名称的由来。
		- 若转义字符 ESC 也出现在数据中，则解决方法仍是在转义字符前插入一个转义字符。
		- 即有特定字符在，就在其后面添加ESC
## 3.2.3 零比特填充法
### 1. 定义
- 零比特填充法<span style="background:rgba(240, 200, 0, 0.2)">允许数据帧包含任意个数的比特,</span> 它使用一个**特定的比特串 01111110** 来**标志一帧的开始和结束**
### 2. 如何实现
![[Pasted image 20240425133458.png|450]]
- 如图 3.7 所示。
- 发送方
	- 为了不使数据字段中出现的比特流 01111110 被误判为帧的首尾标志，发送方**先扫描整个数据字段**，
	- <span style="background:rgba(240, 200, 0, 0.2)">每遇到 5 个连续的“1”，就自动在其后插入一个“0”</span>。
		- 逢 5 加 0
	- 经过这种比特填充后，就可保证数据字段中**不会出现 6 个连续的“1”**。
- 接收方
	- 接收方做该过程的逆操作，即每收到 5 个连续的“1”，就自动删除后面紧跟的“0”，以恢复原始数据。
		- 逢 5 删 0
  
- 在数据链路层早期使用的 **HDLC 协议**中，便是采用这种**比特填充的首尾标志法**来实现透明传输的。
	- ![[2 第三章 数据链路层#4. HDLC 帧格式以及如何帧同步]]
### 3. 优点
- 零比特填充法很容易由硬件来实现，性能优于字节填充法。
## 3.2.4 违规编码法
### 1. 介绍
- 在**物理层**进行比特编码时，常采用违规编码法。
	- 例如
		- 曼彻斯特编码方法
			- 将数据比特“1”编码成“高-低”电平对，
			- 将数据比特“0”编码成“低-高”电平对，
			- 而“高-高”电平对和“低-低”电平对在**数据比特**中是**违规**的 (即**没有采用**)，
			- 因此可借用这些**违规编码序列**来<font color="#ff0000">定界帧的起始和终止</font>。
			  
	- 局域网 **IEEE 802 标准**就采用了这种方法。
### 2. 优点
- 违规编码法**不采用任何填充技术**便能实现数据的透明传输，但**只适用于<font color="#ff0000">采用冗余编码</font>**的特殊编码环境。
### 3. 目前常见的组帧方法
- 因为**字符计数法**中计数字段的**脆弱性**和**字节填充法**实现上的**复杂性与不兼容性**
	- 所以目前较常用的组帧方法是
		- <font color="#ff0000">零比特填充法</font>
		- <font color="#ff0000">违规编码法</font>

|       | 不足       |     | 常见组帧   |
| ----- | -------- | --- | ------ |
| 字符计数法 | 脆弱性      |     | 零比特填充法 |
| 字节填充法 | 复杂性、不兼容性 |     | 违规编码法  |

# 3.3 差错控制
## 3.3.0 前言
### 一、比特差错
- 比特差错
	- 实际**通信链路**都不是理想的，**比特在传输过程中可能产生差错**，1 可能变成 0，0 也可能变成 1，这就是比特差错。
	- 比特差错是传输差错中的一种
### 二、两类编码技术
- 利用编码技术进行差错控制，主要有两类
	- 自动重传请求 (Automatic Repeat reQuest, ARO) 
		- 在 ARO 方式中，当接收方检测到差错时就设法通知发送方重发，直到收到正确的数据为止。
		- 检测差错通知重传，直至正确
		  
	- 前向纠错 (Forward Error Correction，FEC)。
		- 在 FEC 方式中，接收方不但能发现差错，而目能确定错误的位置并加以纠正。
		- 不仅能差错还能准确纠错
### 三、差错控制的分类
- 差错控制又可分为
	- 检错编码
	- 纠错编码

## 3.3.1 检错编码
### 一、检错编码的核心思想
- 检错编码都采用**冗余编码**技术
  
- 核心思想是
	- 在有效数据 (信息位) 被发送前，按某种关系**附加一定的冗余位**，构成一个符合某一规则的码字后发送。
	- 当要发送的有效数据变化时，相应的冗余位也随之变化，使得**码字遵从不变**的规则。（<span style="background:rgba(240, 200, 0, 0.2)">会随有效位变化而变化</span>）
	- 接收方根据收到的码字是否仍符合原规则来判断是否出错。
	  
- 常见的检错编码有
	- <font color="#ff0000">奇偶检验码</font>
	- <font color="#ff0000">循环冗余码</font>
### 二、两种检错编码
#### 1. 奇偶检错码
##### 1. 基本介绍以及组成
- 奇偶检验码是**奇检验码**和**偶检验码**的统称，是一种**最基本的检错码**。
  
- 它由 n-1 位数据和 1 位检验位组成，检验位的取值 (0 或 1) 将使整个检验码中“1”的个数为奇数或偶数。
##### 2. 奇偶检验码
- 1) 奇检验码: 
	- 附加一个检验位后，码长为"的码字中<font color="#ff0000">“1”的个数为奇数</font>。
- 2) 偶检验码: 
	- 附加一个检验位后，码长为 n 的码字中<font color="#ff0000">“1”的个数为偶数</font>。

- 例如，
	- 7 位数据 1001101 
		- 对应的奇检验码为 10011011
		- 对应的偶检验码为 10011010
		   
- 局限
	- 它**只能检测奇数位的出错情况**
	- 但**不知道哪些位错**了
	- 也**不能发现偶数位的出错情况**
#### 2. 循环冗余码
##### 1. 影响
- 数据链路层**广泛使用**循环冗余码 (Cyclic Redundancy Code，CRC) 检错技术。
##### 2. 基本思想
- 1. 收发双方约定生成多项式 G (x)(最高位和最低位必须为 1)。K 位位串可视为阶数为 k-1 的多项式的系数序列。例如，可用多项式 x+x+1 表示位串 1101。
- 2. 发送方基于待发送的数据和 G (x)，计算出冗余码，**将冗余码附加到数据后面一起发送**
- 3. 接收方收到数据和冗余码后，通过 **G (x)** 来**计算**收到的数据和冗余码**是否产生差错**。

例如：
- 假设一个待传送 m 位的数据，CRC 运算产生一个r位的冗余码，称为**帧检验序列 (FCS)。**
- 这样形成的帧将由 m+r 位组成。
- 在所要发送的数据后面增加r位冗余码，虽然增大了传输开销, 但是可以进行差错检测，这种代价往往是值得的。
	- 虽然增大传输开销，但代价值得
  
- 这个**带检验码的帧**刚好**能被预先确定的多项式 G (x) 整除**。
- 接收方用相同的多项式去除收到的帧，若**无余数，则认为无差错**。
##### 3. 循环冗余码的计算
假设一段 m 位数据，则计算冗余码的步骤如下:
- 加 0。
	- 假设 G (x) 的阶为 r，在数据后面加 r 个 0，相当于乘以 2<sup>r</sup>
- 模 2 除。
	- 利用模 2 除法，用 G (x) 对应的二进制串去除 1) 中计算得出的数据串，得到的余数即为冗余码 (共 r 位，前面的 0 不可省略)。
	- 按照模 2 运算规则，加法不进位，减法不借位，相当于对应位进行**逻辑异或运算**。

冗余码的计算举例
- 假设数据 M=101001 (即 m=6)，除数 G (x)=1101 (即 r=3)，
- 经模 2 除法运算后的结果是: 商 0=110101 (这个商没什么用)，余数 R=001。
- 因此，发送出去的数据为 101001 001 (即 2'M +FCS)，共有 m+r 位，
- 运算过程如图 3.8 所示。

##### 4. 冗余码
- 发送方的 FCS 生成和接收方的 CRC 检验都是由**硬件实现**的，**处理非常迅速**，**不会影响数据的传输**。
	- 硬件实现、处理迅速、不影响数据传输
	  
- 若在传输过程中无差错，则经过 CRC 检验后得出的余数 R 肯定为 0。
- 但是，若**出现误码**, 则**余数 R 仍为 0 的概率极低**。
- 因此，通过 CRC 检错技术，可以近似地认为“**凡是接收方数据链路层接受的帧均无差错**”。
	- 接受即无错
  
- 也就是说，
	- 凡是接收方数据链路层接受的帧，我们都能**以非常接近 1 的概率**认为这些帧在传输过程中**未产生差错**; 
	- 而接收方丢弃的帧**虽然曾经收到**，但最终因为有差错而被丢弃，即**未被接受**。
##### 5. 注意
- 循环冗余码 (CRC) 是**具有纠错功能的**
- 只是**<font color="#ff0000">数据链路层仅使用了它的检错功能</font>**，检测到帧出错则直接丢弃，是为了方便协议的实现，
- 因此本节将 CRC 放在检错编码中介绍。

# 3.4 流量控制与可靠传输机制
>在数据链路层中，流量控制机制和可靠传输机制是交织在一起的。
## 3.4.1 流量控制与滑动窗口机制
### 一、流量控制的定义以及方法
- <font color="#ff0000">流量控制</font>是指
	- 由**接收方控制发送方**的**发送速率**，使接收方有足够的缓冲空间来接收每个帧。
- 常见的流量控制方法有两种
	- 停止-等待协议
	- 滑动窗口协议
### 二、数据链路层与传输层的区别（流量控制）
- 数据链路层和传输层均有流量控制的功能，它们都用到了滑动窗口协议，但也有所区别，主要体现如下:

|      | 数据链路层        |     | 传输层                       |
| ---- | ------------ | --- | ------------------------- |
| 控制的是 | 相邻节点之间的流量    |     | 端到端的流量                    |
| 控制手段 | 接收方收不下就不返回确认 |     | 接收方通过确认报文段的窗口值来调整发送方的发送窗口 |

### 三、流量控制的两种方式
#### 1. 停止-等待流量控制的基本原理
- 停止-等待流量控制是一种**最简单**的流量控制方法。
##### 1. 原理
-  发送方每次只允许发送一个帧
- 接收方每接收一个帧都要反馈一个应答信号, 表示可以接收下一帧,
- 发送方收到应答信号后才能发送下一帧
- 若发送方没有收到接收方反馈的应答信号，则需要一直等待
##### 2. 缺点
- 发送方每发送完一个帧，就进入等待接收方确认信息的过程中，因而**传输效率很低。**
#### 2. 滑动窗口流量控制基本原理
- 滑动窗口流量控制是一种**更高效**的流量控制方法。
##### 1. 发送窗口、接收窗口
- 发送窗口
	- 在任意时刻，发送方都维持**一组连续的允许发送帧**的**序号**——称为**发送窗口**
	- 发送窗口表示
		- 在还未收到对方确认信息的情况下，发送方**最多还能发送多少个帧**和**哪些帧**。
- 接收窗口
	- 同时接收方也维持**一组连续的允许接收帧**的**序号**——称为**接收窗口**。
	- 接收方设置接收窗口是
		- 为了<span style="background:rgba(240, 200, 0, 0.2)">控制可以接收哪些帧和不可以接收哪些帧</span>。
##### 2. 工作原理
###### 1. 发送窗口的工作原理
![[Pasted image 20240425215155.png|475]]
- 发送方每收到一个按序确认的确认帧，就将发送窗口向前滑动一个位置。
  
- 这样，就有一个新的序号落入发送窗口，序号落入发送窗口内的数据帧可以继续发送。
  
- 当窗口内没有可以发送的帧 (即窗口内的帧全部是已发送但未收到确认的帧) 时，发送方就停止发送。
###### 2. 接收窗口的工作原理
![[Pasted image 20240425215227.png|475]]
- 接收方每收到一个序号落入接收窗口的数据帧，就允许将该帧收下
- 然后将接收窗口向前滑动一个位置，并发回确认。
- 这样，就有一个新的序号落入接收窗口，序号落入接收窗口内的数据帧即为准备接收的帧。
- 若收到的帧落在接收窗口之外，则一律丢弃。

##### 3. 滑动窗口的重要特性
1. 只有接收窗口向前滑动 (同时接收方发送了确认) 时，发送窗口才有可能 (只有发送方收到确认后才一定) 向前滑动。
   
2. 从滑动窗口的概念看，停止-等待协议、后退 N 帧协议和选择重传协议只在发送窗口大小与接收窗口大小上有所差别:
   ![[Pasted image 20240425215346.png]]
   
3. 当**接收窗口的大小为 1** 时，可保证帧的**有序接收**。
   
4. 在数据链路层的滑动窗口协议中，窗口大小在传输过程中是**固定**的 (<font color="#ff0000">与传输层不同</font>)。

## 3.4.2 可靠传输机制
### 一、可靠传输的概述
#### 1. 基本概念
- 可靠传输是指<span style="background:rgba(240, 200, 0, 0.2)">发送方发送的数据都能被接收方正确地接收</span>，通常采用<font color="#ff0000">确认</font>和<font color="#ff0000">超时重传</font>两种机制来实现。
	- 确认是指
		- 接收方每收到发送方发来的数据帧，都要向发送方发回一个确认帧，表示已正确地收到该数据帧。
		- 逐帧确认
	- 超时重传是指
		- 发送方在发送一个数据帧后就启动一个计时器
		- 若在规定时间内没有收到所发送数据帧的确认帧，则重发该数据帧，直到发送成功为止。
#### 2. 自动重传请求（ARQ）
- <span style="background:#d3f8b6">使用这两种机制的可靠传输协议</span>称为——自动重传请求 (ARQ)，
	- 它意味着重传是自动进行的，接收方不需要对发送方发出重传请求。
	  
- 在 ARO 协议中，**数据帧和确认帧都必须编号**，
	- 以**区分**确认帧是对哪个帧的确认，
	- 以及哪些帧还**未确认**。
  
- ARQ 协议分为三种: 
	- <span style="background:#d3f8b6">停止-等待 (Stop-and-Wait) 协议</span>
	- <span style="background:#d3f8b6">后退 N 帧 (Go-Back-N) 协议</span>
	- <span style="background:#d3f8b6">选择重传 (Selective Repeat) 协议</span>
	- 这三种可靠传输协议的基本原理并**不仅限于数据链路层**，还可应用到其上各层。

#### 3. 数据链路层的可靠性
- 在有线网络中，
	- 链路的误码率较低，
	- 为了降低开销，并不要求数据链路层向其上层提供可靠传输服务，
	- 即使出现了误码，可靠传输的问题也由其上层处理。
- 而无线网络
	- 链路易受干扰，误码率较高，
	- 因此要求数据链路层必须向其上层提供可靠传输服务。

|      | 链路的误码率    | 是否要求数据链路层提供可靠传输服务 | 可靠传输由谁负责 |
| ---- | --------- | ----------------- | -------- |
| 有线网络 | 误码率低      | 不要求               | 上层       |
| 无线网络 | 易受干扰、误码率高 | 要求                | 本层       |

### 二、三种ARQ 协议

#### 1. 单帧滑动窗口与停止-等待协议 (S-W)
##### 一、工作原理
- 在停止-等待协议中，
	- 发送方每次只能发送一个帧，当发送方收到接收方的确认帧之后，才可以发送下一个帧。
- 从滑动窗口的角度看，
	- 停止-等待协议的<font color="#ff0000">发送窗口和接收窗口大小均为 1。</font>
##### 二、三种差错
1. <span style="background:rgba(173, 239, 239, 0.55)">数据帧丢失</span>
   
2. <span style="background:rgba(173, 239, 239, 0.55)">到达接收方的数据帧可能已遭破坏</span>，接收方利用前面介绍的差错检测技术检出后，**简单地将该帧丢弃。**
   - 为了应付这种可能的情况，发送方装备了**计时器**。
	   - 在一个帧发送后，发送方等待确认，
	   - 当计时器超时的时候，若仍未收到确认，则重发该数据帧。
	   - 如此重复，直到该数据帧正确到达为止。
	    
3. <span style="background:rgba(173, 239, 239, 0.55)">数据帧正确而确认帧被破坏</span>
   - 此时接收方已收到正确的数据帧，但发送方收不到确认帧，
   - 因此发送方会重传已被接收的数据帧，接收方收到相同的数据帧时会丢弃该帧，并重传一个该帧对应的确认帧。


##### 三、比特编号数目
- 对于停止-等待协议，因为每发送一个数据帧就停止并等待，所以**用 1 比特来编号就已足够**
	- 发送的帧交替地用 0 和 1 来标识, 
	- 确认帧分别用 ACK 0 和 ACK 1 来表示, 当收到的确认帧有误时，就重传已发送的数据帧。
	  
	  出错判断：
	- 若连续出现相同序号的**数据帧**，则表明**发送方进行了超时重传**。
	- 若连续出现相同序号的**确认帧**，则表明**接收方收到了重复帧**。
##### 四、帧缓冲区
- 此外，**为了超时重传和判定重复帧的需要**，发送方和接收方**都要设置**一个缓冲区。
	- 当发送方发送完数据帧时，
	- 必须在其发送缓存中保留该数据帧的副本, 
	- 这样才能在出现差错时进行重传。
	- <span style="background:#d3f8b6">只有在收到对方发来的确认帧 ACK 后，方可清除该副本</span>。
##### 五、缺点
- <span style="background:#fdbfff">停止-等待协议的信道利用率很低</span>。
	- 为了提高传输效率，
		- 产生了<span style="background:#d3f8b6">连续 ARQ 协议 </span>(后退 N 帧协议和选择重传协议)，
		- 即**<font color="#ff0000">发送方可连续发送多个帧</font>**，而不是每发完一个就停止等待确认。

#### 2. 多帧滑动窗口与后退N 帧协议 (GBN)
##### 一、工作原理
- 在后退 N 帧协议中，发送方可在未收到确认帧的情况下, 将序号在发送窗口内的多个数据帧全部发送出去。
	- 后退 N 帧的含义是: 
		- 发送方发送 N 个数据帧后，
		- 若发现这 N 个帧的前一个数据帧在计时器超时的时候仍未收到其确认信息，则该帧被判为出错或丢失，
			- <span style="background:rgba(173, 239, 239, 0.55)">一个帧出错就要重传该帧以及之后的N帧</span>
		- 此时发送方不得不**重传该出错帧及随后的 N 个帧**。
		- 这意味着，<font color="#ff0000">接收方只允许按顺序接收帧</font>。
##### 二、工作过程
![[Pasted image 20240425223229.png|450]]
- 发送方向接收方发送数据帧
	- 发送方发完 0 号帧后，可以继续发送后续的 1 号帧、2 号帧等。
	- 发送方每发送完一个数据帧，就要为该帧设置超时计时器。
	- 因为连续发送了许多帧，所以确认帧必须指明是对哪个帧的确认。

- 为了降低开销，GBN 协议**允许接收方**进行**累积确认**，
	- 即允许**接收方**<span style="background:rgba(173, 239, 239, 0.55)">不需要每收到一个正确的数据帧就立即发回一个确认帧</span>，
	- 而可在<span style="background:#fff88f">连续收到多个正确的数据帧后，对最后一个数据帧发回确认信息</span>，
		- 也就是说，**对某个数据的确认**就**代表该数据帧和之前所有的帧均已正确无误地收到**。
			- ACKn 表示对 n 号帧的确认，<font color="#ff0000">表示接收方已正确收到 n 号帧及之前的所有帧</font>，下次期望收到 n+1 号帧 (也可能是 0 号帧)。

- 接收方只按序接收数据帧。
	- 图中，虽然在有差错的 2 号帧之后接着收到了正确的 6 个数据帧，但接收方必须将这些帧丢弃。
	- 接收方**虽然丢弃了这些未按序出现的无差错帧**，但**应重发已发送的最后一个确认帧 ACK 1** 
		- (这是为了防止已发送的确认帧 ACK 1 丢失)。
##### 三、GBN 发送窗口的意义/最大尺寸
- 若采用 n 比特对帧编号
	- 则其发送窗口应满足 ![[Pasted image 20240425222615.png|99]]
		- 若 W<sub>T</sub>大于 2<sup>n</sup>-1，则会造成接收方无法分辨新数据帧和旧数据帧
	- 后退 N 帧协议的接收窗口 W<sub>R</sub>=1，可保证按序接收数据帧。

##### 四、后退N 帧协议的缺点
 - 不难看出，后退 N 帧协议
	 - 一方面因连续发送数据帧而提高了信道利用率
	   
	 - 另一方面在重传时**又必须重传原来已正确到达的帧** (仅因这些帧的前面有一帧出错)，
	   
	 - 因此这种做法会**降低传送效率**。
	   
	 - 当信道**误码率较大**时，后退 N 帧协议**不一定优于停止-等待协议**。
#### 3. 多帧滑动窗口与选择重传协议（SR）
##### 一、产生背景
- 为了进一步提高信道的利用率，
	- 可以设法<span style="background:#fff88f">只重传出现差错和计时器超时的数据帧，</span>
		- 只处理出错的帧
	- 但此时必须**加大接收窗口**，以便先收下失序但正确到达且序号仍落在接收窗口内的那些数据帧，
		- 加大接收窗口，让一部分正确的帧有个地方放
	- <span style="background:#fff88f">等到所缺序号的数据帧收齐后，再一并送交上层</span>。这就是**选择重传协议**。
##### 二、工作原理
- 为了使发送方仅重传出错的帧，接收方**不能再采用累积确认**，而要对每个正确接收的数据帧<font color="#ff0000">逐一进行确认</font>。
	- <span style="background:#fff88f">区别后退N 帧协议</span>
	  
- 显然，选择重传协议
	- 比后退 N 帧协议**更复杂**，
	- 且接收方需要**设置足够的帧缓冲区**来**暂存那些失序但正确**到达且序号落在接收窗口内的数据帧。
		-  <span style="background:#fff88f">帧缓冲区的数目等于接收窗口的大小</span>而非序号数目，
		- 因为接收方不能接收序号在窗口下界以下或窗口上界以上的帧)
		  
	- 每个发送缓冲区对应一个计时器，当**计时器超时的时候，缓冲区的帧就重传**。
	  
- 另外，选择重传协议还采用了比上述其他协议**更有效**的**差错处理策略**，
	- 即一旦接收方检测到某个数据帧出错，就**向发送方发送一个否定帧 NAK**，要求发送方立即重传 NAK 指定的数据帧。
	- 在图 3.12 中，
		- 2 号帧丢失后，接收方仍可正常接收并缓存之后收到的数据帧，
		- 待**发送方超时重传 2 号帧**并被接收方成功接收后，接收窗口就可向前移动，
		- 而当发送方收到 2 号帧的确认后，发送窗口就可向前移动。
		  
		- 在某个时刻，接收方检测到 10 号帧出错，向发送方发出**否定帧 NAK 10**，
		- 在此期间接收方仍可正常接收并缓存之后收到的帧，
		- 发送方收到否定帧 NAK 10 后立即重传 10 号帧。
![[Pasted image 20240425225950.png]]
##### 三、窗口满足条件
- 选择重传协议的接收窗口 W<sub>R</sub>和发送窗口 W<sub>T</sub>都大于 1, 一次可以发送或接收多个帧。
- 若采用 n 比特对帧编号，需满足条件
	- ①: W<sub>R</sub>+W<sub>T</sub>≤2<sub>n</sub>
		- 否则，在接收方的接收窗口向前移动后，
		- 若有一个或多个确认帧丢失，
			- 则发送方就会超时重传之前的旧数据帧，
			- 接收窗口内的新序号与之前的旧序号**出现重叠**，
			- 接收方就**无法分辨**是**新数据帧还是重传的旧数据帧**
			  
	- ②:W<sub>R</sub>≤W<sub>T</sub>
		- 否则，若接收窗口大于发送窗口，则接收窗口永远不可能填满，接收窗口多出的空间就毫无意义
		  
	- 由①和②不难得出 W<sub>R</sub>≤2<sup>n-1</sup>
		- 一般情况下，<span style="background:#fff88f">W<sub>T</sub> 和 W<sub>R</sub>的大小是相同的</span>。

## 3.5.1 信道划分介质访问控制
### 一、基本概述
#### 1. 信道划分介质访问控制
- 信道划分介质访问控制
	- 将使用**同一传输介质**的<span style="background:rgba(160, 204, 246, 0.55)">多个设备的通信</span>**隔离**开来，把时域和频域资源合理地分配给这些设备。
	- 通过复用技术实现
		- 所谓复用，是指
			- 在发送端把多个发送方的信号组合在一条物理信道上进行传输，
			- 在接收端把收到的复用信号分离出来，并发送给对应的接收方，

![[Pasted image 20240425234820.png|350]]
如图 3.16 所示。
- 当**传输介质的带宽**<font color="#ff0000">超过</font>**传输单个信号所需的带宽**时，
- 通过在一条介质上传输多个信号，还能**提高传输系统的利用率**
#### 2. 信道划分的实质
信道划分的实质是
- 通过<span style="background:rgba(160, 204, 246, 0.55)">分时、分频、分码</span>等方法，
- 将原来的一个广播信道，**逻辑上**分为几个用于在两个结点之间进行通信的**互不干扰的子信道**，
- 即广播信道————>若干个点对点信道
### 二、信道划分介质访问控制的分类
#### 1. 频分复用（FDM）
##### 1. 概念
- 频分复用 (Frequency Division Multiplexing，FDM) 是指
	- 将信道的**总频带**划分为多个子频带, 
	- 每个子频带作为一个子信道，
	- 每对用户使用一个子信道进行通信，
##### 2. 原理
![[Pasted image 20240425235257.png|350]]
- 所有用户在**同一时间**占用**不同**的**频带资源**。
- 每个子信道分配的频带可不相同，但它们的**总和不能超过信道的总频带**。
- 在实际应用中，
	- 为了**防止子信道之间互相干扰**，**相邻信道间还要加入“隔离频带**”。
##### 3. 优点
- 频分复用的优点在于
	- 充分利用了传输介质的带宽
	- 系统效率较高
	- 实现也较容易
#### 2. 时分复用 (TDM)
##### 1. 概念
- 时分复用 (Time Division Multiplexing，TDM) 是指
	- 将<span style="background:#fff88f">信道的传输时间划分为一段段等长的时间片</span>，称为 **TDM 帧**
	- 每个用户在每个 TDM 帧中占用固定序号的时隙
	  
	- <span style="background:rgba(160, 204, 246, 0.55)">每个用户所占用的时隙周期性地出现</span> (其周期就是 TDM 的长度)
	- 所有用户在**不同的时间**<span style="background:#fff88f">占用同样的信道资源</span>
> [!note] 注意
> - TDM 帧实际上是<font color="#ff0000">一段固定长度的时间</font>，它与数据链路层的帧不是同一个概念。
##### 2. 原理
![[Pasted image 20240425235833.png|450]]
- 从某个时刻来看，
	- 时分复用信道上传送的**仅是某对用户之间的信号**; 
- 从某段时间来看，
	- 传送的是**按时间分割的复用信号**。
	  
- 因为时分复用是**按固定次序给用户分配时隙**的，
	- 当用户在某段时间暂无数据传输时，
	- 其他用户也无法使用这个暂时空闲的线路资源，所以**时分复用后的信道利用率不高**。
##### 3. STDM
- 统计时分复用 (Statistic TDM, STDM) 又称**异步时分复用**, 
	- 它是对 TDM 的一种**改进**。
	- STDM 帧与 TDM 帧不同，它并**不固定分配时隙**，而**按需动态分配时隙**，
	- 当用户有数据要传送时，才会分配到 STDM 帧中的时隙，因此可以**提高线路的利用率**。
		- <span style="background:#fff88f">用到才分配时隙，不用不占时隙</span>

- 例如，
	- 假设线路的数据传输速率为 6000 b/s, 3 个用户的平均速率都为 2000 b/s, 
		- 当采用 TDM 方式时,
			- 每个用户的最高速率为 2000 b/s,
		-  而在 STDM 方式下
			- 每个用户的最高速率可达 6000 b/s。
#### 3. 波分复用 (WDM)
- 波分复用 (Wavelength Division Multiplexing，WDM) 即**光的频分复用**，
	- 它在一根光纤中传输多种不同波长 (频率) 的光信号，
	- 因为**波长不同，各路光信号互不干扰**
	- 最后用**光分用器**将**各路波长分解**出来。
	- 因为光波处于频谱的高频段，有很大的带宽，所以可以**实现多路的波分复用**。
#### 4. 码分复用 (CDM)
##### 1. 概念
- 码分复用 (Code Division Multiplexing，CDM) 
	- 采用**不同的编码来区分各路原始信号**的一种复用方式。
	- 与 FDM 和 TDM 不同，它**<font color="#ff0000">既共享信道的频率，又共享时间</font>**。
##### 2. 码分多址
更常用的名词是码分多址 (Code Division Multiple Access，CDMA)，
- 其原理是将每个比特时间再划分成 m 个短的时间槽，称为码片 (Chip)，
	- 通常 m 的值是 64 或 128，

下例中为简单起见，设 m 为 8。
- 每个站点被指派一个唯一的 m 位码片序列。
	- 发送 1 时，站点发送它的码片序列;
	-  发送 0 时，站点发送该码片序列的反码。
- 当两个或多个站点**同时发送**时，各路数据在信道中**线性相加**。
- 为了从信道中**分离**出各路信号，要求各个站点的码片序列**相互正交**。

简单理解就是
- A 站向 C 站发出的信号用一个向量来表示, 
- B 站向 C 站发出的信号用另一个向量来表示，**两个向量要求相互正交**。
	- <span style="background:#d3f8b6">向量中的分量，就是所谓的码片</span>。
##### 3.  CDMA 的原理
下面举例说明 CDMA 的原理
- 令向量 S 表示 A 站的码片向量，T 表示 B 站的码片向量。
	- 假设 A 站的码片序列被指派为 00011011，
	- 则 A 站发送 00011011 就表示发送比特 1，发送 11100100 就表示发送比特 0。
	- 为了方便计算，**<font color="#ff0000">将码片中的 0 写为-1，将 1 写为+1</font>**，因此 A 站的码片序列是 (-1-1-1 +1+1-1 +1+1)

##### 5. 理解频分复用、时分复用和码分复用
- 假设 A 站要向 C 站运输黃豆，B 站要向 C站运输绿豆，A 站和 B 站与 C 站之间有一条公共的道路，可类比为广播信道。
- 在频分复用方式下，
	- 公共道路被划分为两个车道，分别提供给 A 站到 C 站的车和 B 站到 C 站的车通行，
	- 两类车可同时通行，但都只分到了公共车道的一半，
	- 因此频分复用 (波分复用也一样) 共享时间而不共享空间。
- 在时分复用方式下，
	- 先让 A 站到 C 站的车走一趟，再让 B 站到 C 站的车走一趟，
	- 两类车交替地使用公共车道，
	- 因此时分复用共享空间，但不共享时间。
- 码分复用与另外两种信道划分方式极为不同，在这种方式下，
	- 黄豆与绿豆放在同一辆车上运送，到达 C 站后，由 C 站负责把车上的黄豆和绿豆分开
	- 因此码分复用既共享空间，又共享时间。

|            | 是否共享时间 |     | 是否共享空间 |
| ---------- | ------ | --- | ------ |
| 频分复用（FDM）  | 共享     |     | 不共享    |
| 时分复用 (TDM) | 不共享    |     | 共享     |
| 码分复用 (CDM) | 共享     |     | 共享     |

##### 6. 码分复用技术的优点
- 码分复用技术具有以下优点
	- 频谱利用率高
	- 抗干扰能力强
	- 保密性强
	- 语音质量好
	- 还可以减少投资及降低运行成本
	  
	- 主要用于**无线通信系统**，特别是**移动通信系统**
## 3.5.2 随机访问介质访问控制
### 一、基本概述
- 在随机访问协议中，
	- 不采用集中控制方式解决发送信息的次序问题，所有用户都能根据自己的意愿随机地发送信息，占用信道的全部速率。
#### 1. 碰撞
- 在总线形网络中，当有两个或多个用户同时发送信息时，就会产生帧冲突 (也称碰撞)，**导致所有冲突用户的发送均以失败告终。**
  #### 2. 如何解决
- 为了解决随机访问发生的冲突，
	- <span style="background:#fff88f">每个用户需要按照一定的规则反复地重传它的帧</span>，**直到该帧无冲突地通过**，
	- 这些规则就是随机访问介质访问控制协议，
- 其核心思想是: 
	- 胜利者**通过争用获得信道**，进而**获得信息的发送权**。
	- 因此，随机访问介质访问控制协议又称**争用型协议**。
#### 3. 区别信道划分介质访问

|          | 结点之间的通信                  |
| -------- | ------------------------ |
| 信道划分机制   | 要么共享空间，要么共享时间，要么共享空间和时间; |
| 随机访问控制机制 | 既不共享时间，又不共享空间            |
- 随机介质访问控制实质上是**一种将广播信道转换为点到点信道的机制**
	- 广播信道————>点对点信道
	  
### 二、ALOHA 协议
#### 1. 纯 ALOHA 协议
##### 1. 基本思想
- 纯 ALOHA 协议的基本思想是
	- 当**总线形**网络中的任何站点**需要发送数据**时，<span style="background:#fff88f">可以不进行任何检测就发送数据</span>。
	- 若在**一段时间内未收到确认**，则该站点就认为传输过程中发生了冲突。
	- <span style="background:#fff88f">发送站点需要等待一段时间后再发送数据</span>，直至发送成功。
##### 2. 工作原理
![[Pasted image 20240427210900.png|325]]
图 3.19 表示一个纯 ALOHA 协议的工作原理。
- 每个站均可自由地发送数据帧，假定所有**帧都是定长**的，帧长不用比特而用发送这个帧所需的时间来表示，
	- 图中用T<sub>0</sub>表示这段时间。

- 在图 3.19 的例子中，
	- 当站 1 发送帧 1 时，其他站都未发送数据，所以站 1 的发送必定是成功的。
	  
	- 但随后站 2 和站 N-1 发送的帧 2 和帧 3 在时间上重叠了一部分 (即发生了冲突)。
		- 发生冲突的各站都必须进行重传，但并不能马上进行重传，因为这样做必然导致继续发生冲突。
	- 因此，让各站**等待一段随机的时间**，然后进行重传。
	- 若再次发生冲突，则需要再等待一段随机的时间，直到重传成功为止。
	  
	- 图中其余一些帧的发送情况是，帧 4 发送成功，而帧 5 和帧 6 发生冲突。

##### 3. 缺点
- 纯 ALOHA 网络的**吞吐量很低**，为了克服这个缺点，便产生了时隙 ALOHA 协议。

#### 2. 时隙 ALOHA 协议
##### 1. 基本思想与优化
 - 时隙 ALOHA 协议
	 - <span style="background:#fff88f">同步各站点的时间</span>
	 - 将时间**划分为一段段等长的时隙** (Slot)
	 - 规定站点<span style="background:#fff88f">只能在每个时隙开始时才能发送帧</span>，发送一帧的时间必须**小于或等于**时隙的长度。
	   
	 - 这样做
		 - **避免了用户发送数据的随意性**
		 - **降低了产生冲突的可能性**
		 - **提高了信道的利用率**

### 三、CSMA 协议
#### 一、在 ALOHA 协议上的改进
- ALOHA 网络发生冲突的概率很大。
	- 若每个站点在发送前<font color="#ff0000">都先监听公用信道，发现信道空闲后再发送,</font>
	- 则会大大降低冲突的可能性,从而提高信道的利用率,
	- <span style="background:#fff88f">载波监听多路访问 (Carrier SenseMultiple Access，CSMA) 协议</span>依据的正是这一思想。
	  
- CSMA 协议是在 ALOHA 协议基础上提出的一种**改进协议**
	- 它与 ALOHA 协议的**主要区别**是<font color="#ff0000">多了一个载波监听装置</font>
#### 二、分类以及比较
> 根据**监听方式**和**监听到信道忙后的处理方式**的不同，CSMA 协议分为三种
##### 1. 1-坚持CSMA
- 1-坚持 CSMA 的基本思想是: 
	- 当站点要发送数据时，首先监听信道——先监听
	- 若信道空闲，则立即发送数据——空闲就发送数据
	- 若信道忙，则继续监听直至信道空闲——忙一直监听直到空闲
	  
	- 坚持”的含义是
		- 监听到信道忙时，继续坚持监听信道:
	- “1”的含义是
		- 监听到信道空闲时，立即发送帧的概率为 1。
##### 2. 非坚持CSMA
- 非坚持 CSMA 的基本思想是: 
	- 当站点要发送数据时，首先监听信道——先监听
	- 若信道空闲，则立即发送数据——空闲就发送数据
	- 若信道忙，则放弃监听，等待一个随机的时间后，再重新监听。——忙则放弃监听，随机时间重新监听

- 非坚持 CSMA 协议
	- 在**监听到信道忙时就放弃监听**, 
		- 因此降低了多个站点等待信道空闲后同时发送数据导致冲突的概率
		- 但也**增加了数据在网络中的平均时延**。

##### 3. p-坚持CSMA
- P-坚持 CSMA <font color="#ff0000">只适用于时分信道</font>
- 基本思想是: 
	- 当站点要发送数据时，首先监听信道; 
	- 若信道忙，则持续监听 (即等到下一个时隙再监听)，直至信道空闲; 
	- 若信道空闲，
		- 则以概率 p 发送数据，
		- 以概率 1-p 推迟到下一个时隙再继续监听; 
	- 直到数据发送成功。

- P-坚持 CSMA 检测到信道空闲后，<span style="background:#affad1">以概率 p 发送数据，以概率 1-p 推迟到下一个时隙继续监听</span>，
	- 目的是降低 1-坚持 CSMA 中多个站点检测到信道空闲时同时发送帧的冲突概率;
		- <span style="background:#fff88f">弥补 1-坚持 CSMA 的缺点</span>

- 采用<span style="background:#affad1">坚持“监听”</span>的目的是，
	- 克服非坚持 CSMA 中因随机等待造成的延迟时间较长的缺点。
		- <span style="background:#fff88f">弥补非坚持CSMA 的缺点</span>
	- 因此，p 坚持 CSMA 协议是非坚持 CSMA 协议和 1-坚持 CSMA 协议的**折中**。

### 四、CSMA/CD 协议（载波监听多路访问/冲突检测 协议）
#### 1. 基本概述
- 载波监听多路访问/冲突检测 (CSMACD) 协议
	- 是 CSMA 协议的改进方案，
	- 适用于**总线形网络**或**半双工网络**环境。
	  
	- 对于全双工网络，
		- 因为全双工采用两条信道，分别用来发送和接收，
		- 在任何时候，发收双方都可以发送或接收数据，不可能产生冲突，所以**不需要 CSMA/CD 协议**。
		- 全双工不需要CSMA/CD 协议

#### 2. 载波监听、冲突检测

##### 一、载波监听
- 载波监听是指
	- 每个站点在**发送前和发送过程中**都**必须不停地检测信道**，
		- 在**发送前**检测信道是为了<font color="#ff0000">获得发送权</font>，
		- 在**发送过程中**检测信道是为了<font color="#ff0000">及时发现发送的数据是否发生冲突</font>。
		  
	- 站点要在发送数据前**先监听信道**，<span style="background:#fff88f">只有信道空闲时才能发送</span>。

##### 二、冲突检测
- 冲突检测 (Colision Detection) 就是
	- <span style="background:#fff88f">边发送边监听</span>的，
	- 若监听到了冲突，则立即停止数据发送，
		- 监听到冲突立即停止发送
	- 等待一段随机时间后，重新开始尝试发送数据。


#### 3. CSMA/CD 的工作流程
> CSMA/CD 的工作流程可简单地概括为“<span style="background:#fff88f">先听后发，边听边发，冲突停发，随机重发</span>”。

> **电磁波**在总线上的传播速率总是**有限**的。因此，当某时刻发送站**检测到信道空闲**时，**信道不一定空闲**。

![[Pasted image 20240427225344.png|475]]

- 显然, CSMA/CD 中的站不可能同时进行发送和接收，因此<span style="background:#fff88f">采用 CSMA/CD 协议的以太网只能进行半双工通信。</span>
![[Pasted image 20240427225444.png]]

##### 争用期/冲突窗口
> 从图 3.21 不难看出，
> A 站在开始发送数据后最多经过时间 2r(端到端传播时延的 2 倍) 就能知道有没有发生冲突 
> (当 σ→0 时)。

- 把以太网的端到端往返时间 2 r 称为争用期 (又称冲突窗口)。
	- 每个站在自己发送数据后的一小段时间内，存在发生冲突的可能性，
	- 只有<span style="background:#fff88f">经过争用期这段时间还未检测到冲突时</span>，**才能确定**这次发送**不会发生冲突**。

#### 4. CSMA/CD 最短帧长的理解和相关计算 
- 现在考虑一种情况: 
	- 某站发送一个很短的帧，但在**发送完之前并未检测出冲突**。
	- 假定这个帧在继续向前传播到达目的站之前和别的站发送的帧发生了冲突，
	- 因此**目的站**将收到有差错的帧 (当然会把它**丢弃**)。
	- 然而，**发送站却不知道发生了冲突**，因此**不会重传这个帧**

- 为了避免发生这种情况，**以太网**规定了一个**最短帧长** (争用期内可发送的数据长度)。
	- 在**争用期内**若**检测到冲突**，则**站就停止发送**，
	- <span style="background:#fff88f">此时己发送出去的数据一定小于最短帧长</span>，
		- 因此**凡长度小于**这个最短帧长的帧，<span style="background:#fff88f">就都是因为冲突而异常中止的无效帧</span>。

- 最短帧长的计算公式为
	- 最短帧长=总线传播时延 x 数据传输速率 x 2

- 例如
	- <span style="background:#d3f8b6">以太网</span>**<font color="#ff0000">规定 51.2 us 为争用期的长度</font>**。
	  
	- 对于 **10 Mbs** 的<span style="background:#d3f8b6">以太网</span>，
		- 在争用期内可发送 512 bit, 即 64 B。
		- 当以太网发送数据时，若前 64 B 未发生冲突，则后续数据也不会发生冲突 (表示已成功抢占信道)。
		- 换句话说，若发生冲突，则一定在前 64 B。
		- 因为一旦检测到冲突就立即停止发送，所以这时发送出去的数据一定小于 64 B。
		  
	- 于是，<span style="background:#d3f8b6">以太网</span>规定**<font color="#ff0000">最短帧长为 64 B</font>**，
		- <span style="background:#fff88f">凡长度小于 64 B 的帧，就都是因为冲突而异常中止的无效帧，收到这种无效帧时应立即丢弃</span>。
		- 若只发送小于 64 B 的帧，如 40 B 的帧，
			- 则需要**在 MAC 子层**中于<span style="background:#fff88f">数据字段的后面加一个整数字节的填充字段</span>，
			- 以保证以太网的 MAC 帧的长度不小于 64 B。

#### 5. 二进制指数退避算法的应用
##### 1. 算法产生背景
- 一旦发生冲突，参与冲突的两个站点紧接着再次发送就没有意义，若**坚持这样做，则将导致无休止的冲突。**

- CSMA/CD **采用截断二进制指数退避算法**来**确定冲突后重传的时机**，
	- 它让发生冲突的站点在停止发送后，<font color="#ff0000">推迟一个随机的时间</font>**再重新发送**
##### 2. 算法精髓
![[Pasted image 20240427230705.png|500]]
###### 一、基本步骤
- 确定基本退避时间，
	- 一般取 2 倍的总线端到端的传播时延 2 r(即争用期)。
	  
- 从离散的整数集合[0,1,…, (2*-1)]中随机取出一个数，记为r，
	- 重传所需推迟的时间就是 r 倍的争用期，即 2 rτ。
	- 参数 k=min[重传次数, 10]，
		- 可见当重传次数不超过 10 时，参数 k 等于重传次数; 
		- 但当重传次数超过 10 时，k 就不再增大，而一直等于 10。
		  
-  当**重传达 16 次仍不成功**时，说明网络太拥挤，认为该帧永远无法正确发出，**抛弃该帧并向高层报告出错**
###### 二、工作原理

- 假设适配器首次试图传送一帧，且在传送过程中检测到冲突。
	- 第 1 次重传时，
		- k=1，随机数"从整数集合{0,1}中选择，可选的重传推迟时间是 0 或 2。
	- 若再次发生冲突，则第二次重传时，
		- 随机数 r 从整数集合 10,1,2.3}中选择，因此重传推迟时间是在 0,2 z, 4,6 r 这四个时间中随机选取的一个
		- 以此类推。
		  
- 使用截断二进制指数退避算法可<span style="background:#fff88f">使重传需要推迟的平均时间随重传次数的增大而增大</span> (也称**动态退避**)
  
- 因此**能降低发生冲突的概率**，**有利于整个系统的稳定**。


#### 6. CSMA/CD 算法的归纳
##### 1. 准备发送:
- 适配器从网络层获得一个分组，封装成帧，放入适配器的缓存。
##### 2. 检测信道: 
- 若信道空闲，则开始发送该帧: 
- 若信道忙，则持续检测直至信道空闲。
##### 3. 在发送过程中，适配器仍然持续检测信道。
- 这里只有如下两种可能。
	- 发送成功: 
		- 在争用期内一直未检测到冲突，该肯定能发送成功。
	- 发送失败: 
		- 在争用期内检测到冲突，此时立即停止发送，
		- 适配器执行指数退避算法，等待一段随机时间后返回到步骤 2。
		- 若<span style="background:#fff88f">重传 16 次仍不能成功, 则停止重传并向上报错。</span>

### 五、CSMA/CA 协议
#### 1. CSMA/CD 在无线局域网的局限以及原因
- CSMA/CD 协议已成功用于使用有线连接的局域网，
- 但在无线局域网环境下**不能简单地搬用** CSMA/CD 协议，<font color="#ff0000">特别是冲突检测部分</font>，
	- 主要有两个原因:
		- 接收信号的强度往往远小于发送信号的强度，且在无线介质上信号强度的动态变化范围很大，
			- 因此若要实现冲突检测，则硬件上的花费会过大。
		- 在无线通信中，
			- 并非所有站点都能够听见对方，即存在“隐蔽站”问题。
#### 2. 冲突避免
- <span style="background:#d3f8b6"> 802.11 标准</span>定义了**广泛用于无线局域网的 CSMA/CA 协议**，它对 CSMA/CD 协议**进行修改**，
	- 将冲突检测改为冲突避免 (Collision Avoidance，CA)。
	- 冲突检测CD————>冲突避免CA

- “冲突避免”
	- <span style="background:#fff88f">并不是指协议可以完全避免冲突</span>，
	- 而是指协议的设计要**尽量降低冲突**发生的概率。
	  
- 因为 **802.11 无线局域网不使用冲突检测**，
	- **一旦站点开始发送一个帧**，就会**完全发送该帧**，
	- 但**冲突存在时仍发送整个帧** (尤其是长数据帧) 会**严重降低网络的效率**
	- 所以要<span style="background:#fff88f">采用冲突避免技术来降低冲突的概率</span>。

#### 3. 链路层确认/重传 (ARQ) 
- 因为无线信道的**通信质量远不如有线信道**，<原因>
	- 所以 802.11 标准**使用链路层确认/重传 (ARO) 方案**
	- 即站点每通过无线局域网发送完一帧，就要在**收到对方的确认帧后才能继续发送下一帧**。
		- <span style="background:#fff88f">收到回信才能进行下一步</span>
	  
- 可见，802.11 标准无线局域网采用的**停止-等待协议**是一种**可靠传输协议**。

#### 4. 802.11 标准三种帧间间隔 IFS。
##### 一、帧间间隔 (InterFrame Space，IFS)
- 为了**尽量避免冲突**，
	- 802.11 标准规定，
		- 所有站**完成发送**后，<span style="background:#fff88f">必须等待一段很短的时间 (继续监听) 才能发送下一帧</span>。
			- 这段时间称为**帧间间隔** (InterFrame Space，IFS)。
		- 帧间间隔的**长短取决于**该站要发送的**帧的类型**。

##### 二、三种IFS
###### 1. SIFS (短 IFS)
- <font color="#ff0000">最短的 IFS</font>，用来**分隔属于一次对话的各帧**，
- 使用 SIFS 的帧类型
	- 有 ACK 帧、CTS 帧、分片后的数据帧，以及所有回答 AP 探询的帧等。
###### 2. PIFS (点协调 JS):
- <font color="#ff0000">中等长度的 IFS</font>
- 在 PCF 操作中使用。
###### 3. DIFS (分布式协调 IFS):
- <font color="#ff0000">最长的 IFS</font>，
- 用于异步竞争访问的时延。

#### 5. 虚拟载波监听机制
- 802.11 标准还采用了**虚拟载波监听机制**，
	- 即让源站将它要**占用信道的持续时间** (包括目的站发回 ACK 帧所需的时间) **及时通知给所有其他站**，
		- <span style="background:#affad1">告诉大家我会占用多久</span>
	- 以便使所有其他站在这段时间内都停止发送, 这样就**大大减少了冲突的机会**。“
	  
- 虚拟载波监听”
	- 表示**其他站并未监听信道**，而是**因收到了源站的通知才不发送数据**，
		- 有人让我按兵不动
	- 这种效果**就像是其他站都监听了信道**。



#### 6. 退避算法的使用情况
- 当信道从忙态变为空闲时，
	- 任何一个站要发送数据帧，
		- 不仅要等待一个 DIFS 的间隔，
		- 而且要进入争用窗口，
		  
- 计算随机退避时间**以便再次试图访问信道**，因此降低了冲突发生的概率。
  
- <span style="background:#fff88f">当且仅当检测到信道空闲且这个数据帧是要发送的<font color="#ff0000">第一个数据帧</font></span>时，**才不使用退避算法**，
  
- 其他所有情况都必须使用退避算法，
	- 具体为:
		- 在发送第一个帧之前检测到信道忙;
		- 每次重传:
		- 每次成功发送后要发送下一帧。
		- CSMA/CA 的退避算法与 CSMAICD 的稍有不同 (详见相关的教材)。

#### 7. CSMA/CA 算法的归纳
1. 若站点**最初有数据要发送** (而非发送不成功再进行重传)，**且检测到信道空闲**，
   - 那么在**等待时间 DIFS** 后，就**发送整个数据帧**。
     
2. 否则，
   - 站点执行 CSMA/CA 退避算法，选取一个随机退避值。
   - 一旦检测到信道忙，退避计时器就保持不变。
   - <span style="background:#fff88f">只要信道空闲，退避计时器就进行倒计时。</span>
     
3. 当退避计时器减至 0 时 (这时信道只可能是空闲的)，站点就发送整个帧并等待确认。
   
4.  发送站若收到确认，就知道已发送的帧被目的站正确接收。
   - 这时**要发送第二帧**，就要从步骤  2.开始，执行 CSMA/CA 退避算法，随机选定一段退避时间。
   
5. 若发送站在**规定时间** (由重传计时器控制) 内
	- **未收到确认帧 ACK，就必须重传该帧**，再次使用 CSMA/CA 协议争用该信道，直到收到确认，
	- 或经过若干次重传失败后放弃发送。

#### 8. 处理隐蔽站问题: RTS 和 CTS
##### 1. 隐蔽站问题

##### 2. CSMAVCA 协议进行信道预约的方法
- 为了避免该问题，<span style="background:#fff88f">802.11 标准允许发送站对信道进行预约</span>，
	- 预约资源


![[Pasted image 20240427232900.png|311]]

- 如图 3.23 所示。
	- 源站要发送数据帧之前, **先监听信道**,
	  
	-  若信道空闲, 则<font color="#ff0000">等待时间 DIFS</font> 后, 广播一个请求发送<font color="#ff0000"> RTS (Request To Send) 控制帧</font>，
		- RTS它包括
			- 源地址
			- 目的地址
			- <span style="background:#affad1">这次通信所需的持续时间</span>
			  
	- 若 AP 正确收到 RTS 帧，且信道空闲
		- 则<font color="#ff0000">等待时间 SIFS </font>后，向源站发送一个允许发送 <font color="#ff0000">CTS (Clear To Send) 控制帧</font>，
			- 它也包括
				- <span style="background:#affad1">这次通信所需的持续时间</span>，
				  
	- 源站收到 CTS 帧后，<font color="#ff0000">再等待时间 SIFS</font>，就可**发送数据帧**。
	  
	- 若 **AP 正确收到**了源站发来的数据，则<font color="#ff0000">等待时间 SIFS</font> 后就向源站<font color="#ff0000">发送确认帧 ACK</font>。
	  
	- AP 覆盖范围内的**其他站听到 CTS 帧**后，将**在 CTS 帧中指明的时间内<span style="background:#affad1">抑制发送</span>**。

- CTS 帧有两个目的:
	- 给**源站明确的发送许可**;
	- 指示其他站在预约期内**不要发送**。
###### 区分RTS 与CTS 的通信持续时间
- 源站在 RTS 帧中填写的所需占用信道的持续时间
	- 是从 RTS 帧发送完毕后，到目的站最后发送完 ACK 帧为止的时间，
	- 即“SIFS+ CTS+ SIFS+数据帧+SIFS+ ACK”
	  
- 而 AP 在 CTS 帧中填写的所需占用信道的持续时间，
	- 是从 CTS 帧发送完毕，到目的站最后发送完 ACK 帧为止的时间，
	- 即“SIFS+数据帧+ SIFS+ACK”。

###### RTS 与CTS 的优缺点
- 使用 RTS 帧和 CTS 帧
	- 会使网络的通信效率**有所下降**，
	- 但这两种帧都很短，**与数据帧相比开销不算大**。
- 相反，
	- 若不使用这种控制帧，
	- 则一旦发生冲突而导致数据帧重发，浪费的时间会更多。
	- <span style="background:#affad1">信道预约</span><font color="#ff0000">不是强制性规定</font>，**各站可自行决定使用或不使用**。
	- 只有**当数据帧长超过某个数值**时，使用 RTS 帧和 CTS 帧**才比较划算**。


### 六、CSMA/CD 与 CSMA/CA 的区别:


|         | 冲突是否可以检测和避免                                               |     | 传输介质                  |     | 检测信道空闲方式                               |
| ------- | --------------------------------------------------------- | --- | --------------------- | --- | -------------------------------------- |
| CSMA/CD | 可以检测冲突，但无法避免;                                             |     | 总线形以太网                |     | 通过<br>电缆中的电压变化来检测                      |
| CSMA/CA | 发送数据的同时不能检测信道上有无冲突<br>本结点处没有冲突并不意味着在接收结点处就没有冲突<br>只能尽量避免。 |     | 无线局域网 802.11a/b/g/n 等 |     | 采用<br>能量检测<br>载波检测<br>能量载波<br>混合检测三种检测 |
- 总结: 
	- CSMA/CA 
		- 在发送数据帧之前**先广播告知其他站点**
		- 让其他站点在某段时间内不要发送数据帧，以免发生冲突
		  
	- CSMA/CD 
		- 在发送数据帧之前监听
		- 边发送边监听
		- 一旦发生冲突，就立即停止发送



## 3.5.3 轮询访问：令牌传递协议
### 一、轮询访问的定义
- 在轮询访问中，
	- 用户**不能随机**地发送信息
	- 而要**通过**一个**集中控制的监控站**
	- 以**循环方式轮询每个结点**
	- **再决定**信道的分配
  
- 典型的轮询访问控制协议是**令牌传递协议**。
### 二、令牌
- 在令牌传递协议中，
	- 一个令牌 (Token) 沿着环形总线**在各站之间依次传递**。
		- 令牌是一个**特殊的控制帧**，它本身并不包含信息，仅控制信道的使用，确保同一时刻只有一个站独占信道。
		  
- 当环上的一个站希望发送帧时，**必须等待令牌**。
  
- 站点**只有取得令牌后才能发送帧**，因此令牌环网络**不会发生冲突** (因为令牌只有一个)。
  
- 站点**发送完一帧后，应释放令牌**，以便让其他站使用。
  
- 因为**令牌在网环上是按<font color="#ff0000">顺序依次传递</font>**的，所以对所有入网计算机而言，**访问权是公平的**。

### 三、令牌和数据的传递过程
- 当**网络空闲**时，环路中**只有令牌帧在循环传递**。
  
- 当令牌传递到有数据要发送的站点时，
	- 该站点就修改令牌中的一个标志位，
	- 并在令牌中附加自己需要传输的数据，将令牌变成一个数据帧，
	- 然后将这个数据帧发送出去。
	  
- 数据帧沿着环路传输，接收到的站点一边转发数据，一边查看帧的目的地址。
	- 若目的地址和自己的地址相同，则接收站就复制该数据帧，以便进一步处理。
	  
- 数据帧沿着环路传输，直到到达该帧的源站点，源站点收到自己发出去的帧后便不再转发。
	- 同时，通过检验返回的帧来查看数据传输过程中是否出错，若出错则重传。
	  
- 源站点**传送完数据后，重新产生一个令牌**，并传递给下一站点，**交出信道控制权**。

### 四、适用范围以及优点
- 令牌传递协议非常**适合负载很高的广播信道**，
	- 即<span style="background:#fff88f">多个结点在同一时刻发送数据概率很大的信道。</span>
	  
- 若这样的信道<span style="background:#affad1">采用随机介质访问控制，则发生冲突的概率很大</span>
- 而<span style="background:#fff88f">采用令牌传递协议则可以很好地满足各站点间的通信需求</span>。
  
- 令牌传递协议<span style="background:#fff88f">既不共享时间，又不共享空间</span>;
  
-  它**实际上在随机访问介质访问控制的基础上**，<span style="background:#fff88f">限定了有权发送数据的结点只能有一个</span>。

### 五、注意
- 即使是**广播信道**也可**通过介质访问控制机制**，<span style="background:#b1ffff">使广播信道变为逻辑上的点对点信道</span>
  
- 所以说<span style="background:#fff88f">数据链路层研究的是“点到点”之间的通信</span>。

## 3.6.1 局域网的基本概念和体系结构
### 一、局域网的概念
- 局域网 (Local Area Network, LAN) 是指
	- 在一个**较小的地理范围** (如一所学校) 内，
	- 将**各种计算机、外部设备和数据库系统**等通过<span style="background:#fff88f">双绞线、同轴电缆</span>等连接介质互相连接起来，
	- 组成**资源和信息共享**的计算机互连网络。

### 二、局域网的特点
- 为**一个单位所拥有**，且地理范围和站点数目均有限。
- **所有站点共享较高的总带宽** (即较高的数据传输速率)。
- 较低的时延和较低的误码率。
	- 低时延、低误码率
- 各站为**平等关系**而非主从关系。
- 能进行**广播**和**多播**

### 三、局域网的特性、拓扑结构
- 局域网的特性主要由**三个要素**决定: 
	- 拓扑结构
	- 传输介质
	- 介质访问控制方式
	  
	- 其中**最重要的是介质访问控制方式**，它**决定着局域网的技术特性**。

- 常见的局域网拓扑结构主要有以下 4 大类:
	- 星形结构
	- 环形结构
	- 总线形结构
	- 星形和总线形结合的复合型结构。
### 四、局域网的物理组成、介质访问
- 局域网可以使用**铜缆、双绞线和光纤**等多种传输介质，其中<font color="#ff0000">双绞线为主流</font>传输介质。
  
- 局域网的介质访问控制方法主要有 
	- CSMA/CD 协议、
	- 令牌总线协议
	- 令牌环协议
	  
	- 其中**前两种**协议主要用于**总线形局域网**
	- **令牌环协议**主要用于**环形局域网**。

**三种特殊的局域网**拓扑实现如下:
- 以太网 (目前使用范围最广)。逻辑拓扑是总线形结构，物理拓扑是星形结构
- 令牌环 (Token Ring，IEEE 802.5)。逻辑拓扑是环形结构，物理拓扑是星形结构
- FDDI (光纤分布数字接口，IEEE 802.8)。逻辑拓扑是环形结构，物理拓扑是双环结构

|                    | 逻辑拓扑  |     | 物理拓扑 |
| ------------------ | ----- | --- | ---- |
| 以太网                | 总线型结构 |     | 星型结构 |
| 令牌环<br>IEEE 802.5  | 环形结构  |     | 星型结构 |
| FDDI<br>IEEE 802.8 | 环形结构  |     | 双环结构 |

### 五、IEEE 802 标准的局域网
- <span style="background:#d3f8b6">IEEE 802 标准</span>定义的局域网参考模型
	- <span style="background:#fff88f">只对应于 OSI 参考模型的数据链路层和物理层</span>
	  
	- 并将数据链路层拆分为两个子层:
		- <span style="background:#b1ffff">逻辑链路控制 (LLC) 子层</span>
			- 与**接入传输介质有关**的内容都放在 MAC 子层，它**向上层屏蔽对物理层访问的各种差异**，
			- 主要功能包括: 
				- 组帧和拆卸帧
				- 比特传输差错检测
				- 透明传输
		- <span style="background:#b1ffff">介质访问控制 (MAC) 子层</span>。
			- LLC 子层**与传输介质无关**
			- 它向网络层提供
				- 无确认无连接
				- 面向连接
				- 带确认无连接
				- 高速传送
				  
				- 四种不同的连接服务类型。

- 因为在局域网市场中的垄断地位，**<font color="#ff0000">以太网几乎成为局域网的代名词</font>**，
	- 而 802 委员会制定的 **LLC 子层作用已经不大**，所
	- 以**现在**许多网卡**仅装 MAC 协议**而不装 LLC 协议。

## 3.6.2 以太网与 IEEE 802.3
### 一、以太网的发展以及版本协议
- 以太网规约的第一个版本是 DIX V 1，它由 DEC、Intel 和 Xerox 联合提出。
- 之后，它被修改为第二版规约 DIX Ethernet V 2，是世界上第一个局域网产品的规约。
- 在此基础上，IEEE 802 委员会的 IEEE 802.3 工作组制定了第一个 IEEE 的以太网标准 IEEE 802.3。

|                             | 以太网版本            |
| --------------------------- | ---------------- |
| 以太网规约的第一个版本                 | DIX V 1          |
| 第二版规约<br>世界上**第一个局域网**产品的规约 | DIX Ethernet V 2 |
| 第一个 IEEE 的以太网标准             | IEEE 802.3       |


- 以太网是**目前最流行**的**有线局域网技术。**
	- 以太网
		- **逻辑上**采用**总线形**拓扑结构
		- 所有计算机共享同一条总线
		- 信息以广播方式发送
	- 以太网
		- 使<span style="background:#b1ffff">用 CSMA/CD 方式对总线进行访问控制</span>。
  
- 严格来说，**以太网是指符合 DIX Ethernet V 2 标准的局域网**，
	- 但 DIX Ethernet V 2 标准与 IEEE 802.3 标准的差别很小
	- 因此**通常将 802.3 局域网简称为以太网**。

### 二、以太网简化通信的措施
- 以太网采用两项措施来简化通信:
	- 采用**无连接**的工作方式，
		- 既不对发送的数据帧编号，又不要求接收方发送确认，
		- 即以太网**尽最大努力交付数据**，提供的是**不可靠服务**，**对差错的纠正**则由**高层完成**:
		  
	- 发送的数据**都使用**<font color="#ff0000">曼彻斯特编码</font>的信号
		- 每个码元的**中间出现一次电压转换**
		- 接收方利用这种电压转换**方便地将位同步信号提取出来**。

##### 2. 适配器的功能
- <font color="#ff0000">适配器</font>的功能
	- 不仅能**实现与局域网传输介质之间的物理连接**和**电信号匹配**，
	- 涉及
		- 帧的发送与接收
		- 帧的封装与拆封
		- 介质访问控制
		- 数据的编码与解码
		- 数据缓存等功能
		  
	- 当适配器收到正确的帧时，就使用**中断**来通知该计算机，并**交付协议栈中的网络层**。
		- 当计算机要**发送 IP数据报**时, 就**由协议栈把 IP 数据报向下交给适配器**，**组帧后发送到局域网**。
### 四、以太网的MAC 地址
#### 1. MAC 地址的定义
- <span style="background:#d3f8b6">IEEE 802 标准</span>为**局域网**规定了一种 **48 位的全球地址**，
	- 是指局域网上的每台计算机中<font color="#ff0000">固化在适配器的 ROM 中的地址</font>，称为**物理地址**或 **MAC 地址**
		-  (因为这种地址用在 MAC 帧中)
		  
	- 这个地址**用于控制主机在网络上的数据通信**。
		- 全世界所有的局域网适配器都具有不同的地址，一台计算机只要没有更换适配器，不管其地理位置如何变化，其 MAC 地址都不会变化。
#### 2. 以太网 MAC 地址的长度
- MAC 地址长 6 字节,一般用由连字符 (或冒号) 分隔的 12 个十六进制数表示,
	-  如 **02-60-8c**-*e4-b1-21*
		- 高 24 位为厂商代码
		- 低 24 位为厂商**自行分配**的适配器序列号。

- 当**路由器**通过适配器连接到局域网时,
	- 适配器上的 MAC 地址就**用来标志路由器的某个接口**。
- 路由器若**同时连接到两个网络**上，则它**需要两个适配器**和**两个 MAC 地址**
#### 3. 三种帧
- 适配器从网络上每收到一个 MAC 帧，首先都要用硬件检査 MAC 帧中的目的地址。
	- 若是**发往本站的帧**，则收下，否则丢弃该帧。
- 这里“发往本站的帧”包括以下三种帧:
	- 单播帧 (一对一)
		- 即**收到的帧的目的地址**与**本站的 MAC 地址相同**。
	- 广播帧 (一对全体)
		- 即发送给**本局域网上所有站点**的 (全 1 地址)。
	- 多播帧 (一对多)
		- 即发送给**本局域网上一部分站点的帧**。

	- DIX Ethernet V 2 标准 (即以太网 V 2 标准) 
	-  IEEE 802.3 标准
- 这里只介绍最常用的**以太网 V 2 的 MAC 帧格式**，如图 3.25 所示。

- 在帧前面插入的 8 字节前导码分为两个字段: 
	- 第一个字段是 7 字节的**前同步码**，
		- 用来实现 MAC 帧的比特同步; 
	- 第二个字段是 1 字节的**帧开始定界符**，
		- 表示后面的信息就是 MAC 帧。

> [!note] 注意
> - <font color="#ff0000">以太网帧不需要帧结束定界符</font>，
> 	- 因为当以太网传送帧时，各帧之间必须有一定的间隙。
> - 因此，接收方只要找到帧开始定界符，其后面连续到达的比特流就都属于同一个帧。
> - 实际上，
> 	- 以太网采用了**违规编码法**的思想，因为**以太网使用曼彻斯特编码**，所以每个码元中间都有一次电压的跳变。
> 	- 发送方发完一个帧后，发送方网络接口上的电压不再变化，这样接收方就能很容易地找到帧的结束位置，
> 	- 这个位置**往前数 4 字节就是 FCS 字段**，于是就**能确定数据字段的结束位置**。

- 目的地址: 6 字节，帧在局域网上的**目的适配器的 MAC 地址**。
- 源地址: 6 字节，传输帧到局域网上的**源适配器的 MAC 地址**。
- 类型: 2 字节，指出数据字段中的数据应**交给哪个上层协议**处理，
	- 如网络层的 IP 协议。
- 数据:46~1500 字节，承载上层的协议数据单元 (如 IP 数据报)。
  
- 以太网的**<font color="#ff0000">最大传输单</font><font color="#ff0000">**元</font>是 **1500 字节**，
	- 若 IP 数据报超过 1500 字节，则必须将该 IP 数据报**分片**。
	  
- 此外，由于 CSMA/CD 算法的限制，以太网帧必须满足**<font color="#ff0000">最小长度</font>**是 64 字节，
	- 当数据字段的长度**小于 46 字节**时，MAC 子层就在数据字段的后面加一个整数字节的**填充字段**，以确保帧长不小于 64 字节。

## 3.6.3 IEEE 802.11 无线局域网
### 一、无线局域网的组成
- 无线局域网可分为两大类: 
  
	- 有固定基础设施的**无线局域网**
		- 所谓“固定基础设施”，是指**预先建立的、能覆盖一定地理范围的固定基站**。
		  
	- 无固定基础设施的**移动自组织网络**。
#### 1. 有固定基础设施的无线局域网
##### 一、802.11 标准
- 对于有固定基础设施的无线局域网
	- IEEE 制定了无线局域网的 802.11 系列协议标准，
		- 包括 **802.1 la/b/g/mn** 等。
		  
	- 802.11 标准使用**星形**拓扑，其<font color="#ff0000">中心称为接入点 (Access Point，AP)</font>，在 **MAC 层**<font color="#ff0000">使用 CSMA/CA 协议</font>
	  
	- <span style="background:#fff88f">使用 802.11 系列协议的局域网</span>又称 **Wi-Fi**。

##### 二、基本服务集（BSS）
###### 1. 基本概念
- 802.11 标准规定**无线局域网**的**最小构件**是<span style="background:#d3f8b6">基本服务集 (Basic Service Set，Bss)</span>。
	- 一个基本服务集包括**一个接入点**和**若干移动站**。
	  
- 各站在本 BSS 内的通信，或与本 BSS 外部站的通信，都**<font color="#ff0000">必须通过本 BSS 的 AP</font>**。
	- 上面提到的 AP 就是基本服务集中的基站 (base station)。
	- <span style="background:#fff88f">通信必须通过AP</span>
	  
- 安装 AP 时，**必须为其分配**一个不超过 32 字节的**服务集标识符** (Service Set IDentifier，SSID) 和一个**信道**。
	- SSID 是指**使用该 AP 的无线局域网的名称**。
	  
- 基本服务集**覆盖的地理范围**称为<span style="background:#d3f8b6">基本服务区 (Basic ServiceArea，BSA)</span>
	- **无线局域网**的基本服务区的**直径一般不超过 100 m**。

##### 三、分配系统（DS）和扩展的服务集（ESS）
- 基本服务集**可以是孤立**的，
	- 也可**通过 AP 连接到一个分配系统 (Distribution System，Ds)**，
	- **然后连接到另一个基本服务集**，构成一个扩展的服务集 (Extended Service Set，ESS)。

- 分配系统的作用
	- 是使扩展的服务集对上层的表现就像一个基本服务集

- ESS 还可通过一种称为 Portal (门户) 的设备
	- 为**无线用户提供到有线连接**的**以太网接入**。
	- <span style="background:#fff88f">门户的作用相当于网桥</span>。

			- 地址 1 是 B 站的 MAC 地址，

## 3.6.4 VLAN 基本概念与基本原理
>虚拟局域网只是**<span style="background:#fff88f">局域网为用户提供的一种服务</span>**，并不是一种新型局域网。
### 一、虚拟局域网出现的背景
- 一个以太网是一个广播域，当一个以太网中包含的计算机太多时，往往会导致:
	- 以太网中**出现大量的广播帧**，特别是经常使用的 ARP 和 DHCP 协议 (第 4 章)。
	- 一个单位的不同部门**共享一个局域网**，**对信息保密和安全不利**。

- 通过虚拟局域网 (Virtual LAN，VLAN)，
	- 可将一个较大的局域网**分割**成一些较小的与地理位置无关的逻辑上的 VLAN，
	- <span style="background:#d3f8b6">每个 VLAN 是一个较小的广播域</span>。
### 二、三种划分虚拟局域网的方式
#### 1. 基于接口
- 将交换机的若干接口划为一个逻辑组
- 这种方法最简单、最有效
- 若主机离开了原来的接口，则可能进入一个新的子网
#### 2. 基于MAC 地址
- 按 MAC 地址将一些主机划分为一个逻辑子网，
- 当主机的物理位置从一个交换机移动到另一个交换机时，它**仍属于原来的子网**。
#### 3. 基于IP 地址
- 根据网络层地址或协议划分 VLAN，
- 这样的 VLAN **可以跨越路由器**进行扩展，将多个局域网的主机连接在一起。
### 三、VLAN 标签以及VLAN 帧 
#### 1. VLAN 标签
##### 1. 定义
- <font color="#ff0000">802.3 ac 标准</font>定义了支持 VLAN 的**以太网帧格式的扩展**。
	- 它在以太网帧中**插入一个 4 字节的标识符** (插在<span style="background:#fff88f">源地址字段和类型字段之间</span>)，称为 **VLAN 标签**
	- 用来**指明发送该帧的计算机属于哪个虚拟局域网**。
	- <span style="background:#fff88f">虚拟局域网的身份证</span>
#### 2. 802.1Q 帧的格式
> 插入 VLAN 标签的帧称为 802.10 帧，如图 3.30 所示。
![[Pasted image 20240427170348.png|450]]
- 因为 VLAN 帧的首部增加了 4 字节，所以以太网的最大帧长从原来的 1518 字节变为 1522 字节。

- VLAN 标签的**前两个字节**总是置为 **0 x 8100**，表示**这是一个 802.1Q 帧**。
  
- 在 VLAN 标签的后两个字节中，
	- 前 4 位实际上并没什么作用，这里不讨论，
	- <font color="#ff0000">后 12 位</font>是该 VLAN 的**标识符 VID**, 它<font color="#ff0000">唯一地标识该 802.10 帧属于哪个 VLAN</font>。
		- 12 位的 VID 可识别 4096 个不同的 VLAN。
		  
- 插入 VLAN 标签后，802.10 帧最后的 **FCS 必须重新计算**。

#### 3. 工作原理
![[Pasted image 20240427170717.png|475]]
- 如图 3.31 所示，
	- 交换机 1 连接 7 台计算机，该局域网划分为两个虚拟局域网 VLAN-10 和 VLAN-20，
		- 这里的 10 和 20 就是 802.1 Q 帧中的 VID 字段的值，由交换机管理员设定。
			- <span style="background:#fff88f">各主机并不知道自己的 VID 值 (但交换机必须知道)，主机与交换机之间交互的都是标准以太网帧</span>。
				- 交换机知道各个主机的VID 值
				- 主机与交换机交互都是**标准以太网帧**（脱下，穿上）
			- <span style="background:#fff88f">一个 VLAN 的范围可以跨越不同的交换机，**前提是**所用的交换机能够识别和处理 VLAN。</span>

	- 交换机 2 连接 5 台计算机，并与交换机 1 相连。
		- 交换机 2 中的 2 台计算机加入 VLAN-10，另外 3 台计算机加入 VLAN-20。
			- 这**两个 VLAN 虽然都跨越了两个交换机，但各自都是一个广播域**。
			  
- 汇聚链路/干线链路
	- <span style="background:#fff88f">连接两个交换机接口之间的链路</span>称为汇聚链路或千线链路。
	  
- 假定 A 站向 B 站发送帧, 
	- 交换机 1 根据帧首部的目的 MAC 地址,
	- 识别 B 站属于本交换机管理的 VLAN-10，
	- 因此**就像在普通以太网中那样直接转发帧**。
	  
- 假定 A 站向 E 站发送帧，
	- 交换机 1 必须把帧转发到交换机 2，
	- 但在转发前，要插入 VLAN 标签，
	- 否则交换机 2 就不知道应把帧转发给哪个 VLAN。
	- 因此，**在干线链路上传送的帧是 802.1 Q 帧**。
	- 交换机 2 在向 E 站转发帧之前，
	- 要*<span style="background:#fff88f">拿走已插入的 VLAN 标签</span>*，
	- 因此 **E 站收到的帧是 A 站发送的标准以太网帧**，而**不是 802.10 帧**。
	  
- 若 A 站向 C站发送帧，
	- 则情况就复杂了，
	- 因为**这是在不同网络之间的通信**，
	- 虽然 A 站和 C 站**都连接到同一个交换机**，
	- **但是它们已处在不同的网络中** (VLAN-10 和 VLAN-20)，
	- <span style="background:#fff88f">需要通过上层的路由器来解决</span>，
	- 也<span style="background:#fff88f">可在交换机中嵌入专用芯片进行转发</span>，以便**在交换机中实现第 3 层的转发功能**。
# 3.7 广域网
## 3.7.1 广域网的基本概念
### 一、广域网的概念
- 广域网 (Wide Area Network，WAN) 通常是指
	- **覆盖范围很广** (远超一个城市的范围) 的**长距离网络**
	- 任务是**长距离运送主机所发送的数据**
	  
- 连接广域网各结点交换机的链路都是**高速链路**
- 广域网<span style="background:#fff88f">首要考虑的问题</span>是**通信容量必须足够大**，以便支持日益增长的通信量。
### 二、广域网不等于互联网
- 广域网不等于互联网。
	- <span style="background:#fff88f">互联网可以连接不同类型的网络</span>, 通常使用**路由器**来连接。
	- 图 3.32 显示了**由相距较远的局域网**<font color="#ff0000">通过**路由器**</font><font color="#ff0000">与广域网相连</font>而成的一个覆盖范围很广的互联网。
	- 因此，<span style="background:#fff88f">局域网可以通过广域网与另一个相隔很远的局域网通信</span>。

![[Pasted image 20240426005254.png]]
### 三、广域网的组成
- 广域网由**一些结点交换机** 及连接这些交换机的**链路**组成
	-  注意**不是路由器**，
		- 结点交换机和路由器都用来转发分组，它们的工作原理也类似。
			- 结点交换机在**单个网络**中转发分组，
			- 路由器在**多个网络构成的互联网**中转发分组

|       | 在哪转发分组     |
| ----- | ---------- |
| 结点交换机 | 单个网络       |
| 路由器   | 多个网络构成的互联网 |

- 结点交换机的功能是<font color="#ff0000">存储并转发分组</font>。
  
- 结点之间都是**点到点连接**，但为了提高网络的可靠性，通常<span style="background:#fff88f">一个结点交换机往往与多个结点交换机相连</span>。

### 四、广域网与局域网的区别
![[Pasted image 20240426005116.png]]

- 从层次上考虑，
	- 广域网和局域网的区别很大，
		- 因为局域网使用的协议主要在数据链路层 (还有少量在物理层)
		- 而广域网使用的协议主要在网络层。

|     | 使用的协议在哪        |
| --- | -------------- |
| 广域网 | 数据链路层<br>少量物理层 |
| 局域网 | 网络层            |

- 怎么理解“<span style="background:#d3f8b6">局域网使用的协议主要在数据链路层，而广域网使用的协议主要在网络层</span>”这句话呢? 
	- 若**网络中的两个结点**要**进行数据交换,** 
		- 则结点除了要给出数据，
		- 还要给数据“包装”上一层控制信息，用于实现传输控制等功能。
		  
	- 若这层控制信息是
		- 数据链路层协议的控制信息，则称使用了**数据链路层协议**
		- 网络层的控制信息，则称使用了**网络层协议**
### 5. 注意（HDLC）
[[2 第三章 数据链路层#4. HDLC 帧格式以及如何帧同步|第三章  数据链路层]]
- 在**通信线路质量较差**的年代，
	- 能实现**可靠传输**的<span style="background:#fff88f">高级数据链路控制 (HDLC) </span>成为当时比较流行的数据链路层协议。
	  
- 但对现在**误码率很低**的点对点有线链路，
	- 更简单的<span style="background:#fff88f">点对点协议 (PPP)</span> 则是目前使用**最广泛**的数据链路层协议。
- HDLC 已从最新大纲中删除，所以本书不再介绍。
## 3.7.2 PPP 协议
### 一、基本概念
- 点对点协议 (Point-to-Point Protocol，PPP) 是现在**最流行**的**点对点链路控制协议**。
  
- 主要有两种应用:
	- 用户通常都要连接到某个 ISP 才能接入互联网，
		- PPP 协议就是**用户计算机与 ISP 通信**时所用的数据链路层协议
		  
	- 广泛用于**广域网路由器之间**的专用线路。

### 二、三个组成部分
- 一个**链路控制协议 (LCP)**。
	- 用来建立、配置、测试数据链路连接，以及协商一些选项。
	  
- 一套**网络控制协议 (NCP)**。
	- PPP 协议**允许采用多种网络层协议**
	- 每个不同的网络层协议要用一个相应的 NCP 来配置，为网络层协议建立和配置逻辑连接。
	  
-  一种将 **IP数据报封装到串行链路**的方法。
	- IP 数据报在 PPP 帧中就是其**信息部分**，这个信息部分的**长度受最大传送单元 (MTU) 限制**。
### 三、ppp 帧的格式
PPP 帧的格式如图 3.33 所示
![[Pasted image 20240427143541.png|425]]
#### 1. ppp 帧的定界符/标志字段
- PPP 帧的**定界符**
	- 首部和尾部各有一个标志字段 (F)，规定为 0 x 7 E (01111110)，
	- 它表示一个帧的开始和结束, 即 PPP 帧的定界符。
- 当**标志字段**（F）出现在信息段中时，就必须采取一些措施使这种形式上**和标志字段一样的比特组合不出现在信息段**
	- 当 PPP 使用**异步传输**时
		- 采用**字节填充法**，使用的转义字符是 0 x 7 D (01111101)。
	- 当 PPP 使用**同步传输**时
		- 采用**零比特填充法**来实现透明传输。
#### 2. ppp 帧的地址、控制字段
- **地址字段** (A) 占<font color="#ff0000"> 1 字节</font>，规定为 0 xFF，
- **控制字段** (C) 占 <font color="#ff0000">1 字节</font>，规定为 0 x 03，这两个字段的意义暂未定义

- PPP 是**面向字节**的，因此所有 **PPP 帧的长度都是<font color="#ff0000">整数个</font>字节**。
#### 3. ppp 帧的协议段
- 第四个字段是**协议段**，占<font color="#ff0000"> 2 字节</font>，
	- 它表示信息段运载的是什么种类的分组。
		- 若为 **0 x 0021**，则信息字段是 **IP 数据报**。
		- 若为 **0 xC 021**，则信息字段是 **PPP 链路控制协议 (LCP) 的数据**。
#### 4. ppp 帧的信息段
- 第五段**信息段**的长度是可变的，长度为<font color="#ff0000"> 0~1500 字节</font>。
	- 注意
		- 因为 PPP 是**点对点**的，并***不是总线形***，
		- 所以**无须使用 CSMA/CD 协议**，自然就**不会有最短帧长的限制**，
		- 所以信息段占 0~1500 字节，而<font color="#ff0000">不是 46~1500 字节</font>。
#### 5. ppp 帧的帧检验序列
- 第六个字段是**帧检验序列 (FCS)**，占 <font color="#ff0000">2 字节</font>，是使用 **CRC 检验的冗余码**。
### 四、ppp 协议状态图
![[Pasted image 20240427144539.png|475]]
 1) 当处于链路**静止状态**时，通信双方**不存在物理层的连接**。
    
2) 当链路一方检测到载波信号并建立物理连接时，进入**链路建立**状态。
   
3) 在链路建立状态下，**链路控制协议 (LCP)** 开始**协商一些配置**选项 (包括最大帧长、鉴别协议等)。
   - 若协商成功，则进入鉴别状态。
   - 若协商失败，则退回到链路静止状态。
   
4) 协商成功后，双方就建立了 LCP 链路，接着进入“鉴别”状态。
   - 若双方无须鉴别或鉴别身份成功，则进入网络层协议状态。
   - 若鉴别失败，则进入链路终止状态。
   
5) 处于网络层协议状态后，双方**采用 NCP 配置**网络层，<span style="background:#d3f8b6">配置协商成功后，进入链路打开状态</span>。
   
6) <span style="background:#d3f8b6">只要链路处于打开状态</span>，<span style="background:#d3f8b6">双方就可进行数据通信</span>。
   
7) 数据传输结束后，链路一方发出终止请求且在收到对方发来的**终止确认**后，或者**链路出现故障**时，进入链路终止状态。
8) 载波停止后，回到链路静止状态。

![[Pasted image 20240421224429.png]]

### 五、ppp 协议的特点
- PPP 不使用序号和确认机制，只保证无差错接收 (CRC 检验)，因此是**不可靠服务**。
  
- PPP **只支持全双工**的点对点链路，不支持多点线路。
  
- PPP 的**两端可以运行不同的网络层协议**，但仍可使用同一个 PPP 进行通信。
  
- PPP 是**面向字节**的，所有 PPP 帧的长度都是整数个字节。

即只保证无差错传输、不可靠服务。仅全双工点对点链路。两端可以运行不同网络层协议。面向字节。



# 3.8 数据链路层设备
## 3.8.1 网桥的基本概念
### 一、网桥的产生背景
- 使用**集线器**在**物理层**扩展以太网**会形成更大的冲突域**。

- 为了避免这个问题，
	- 可以使用**网桥**在**数据链路层**扩展以太网，而**原来的每个以太网**称为一个网段。
	  
	- 使用网桥进行扩展时，<span style="background:#fff88f">不会将原本独立的两个冲突域合并成一个更大的冲突域</span>，
		- 这是因为**网桥具有识别帧和转发帧**的能力，
		- 根据**帧首部中的目的 MAC 地址**和**网桥的帧转发表**来转发或丢弃所收到的帧, 起到了**过滤通信量**的功能。
		- 因为**各个网段是相对独立**的，所以<span style="background:#fff88f">一个网段的故障不影响另一个网段的运行</span>。

### 二、网桥的连接

- 网络 1 和网络 2 通过网桥连接后，
	- 网桥接收网络 1 发送的数据帧，检查数据中的地址，
		- 若是网络 2 的地址，则转发给网络 2; 
		- 若是网络 1 的地址，则将其丢弃，
			- 因为源站和目的站处在同一个网段，目的站能够直接收到这个帧，而不需要借助网桥转发。
- <font color="#ff0000">异站转发、同站丢弃 </font>
## 3.8.2 以太网交换机
### 一、交换机的原理
- 以太网交换机也称**二层交换机**
	- 二层是指以太网交换机**工作在数据链路层**
	  
- 以太网交换机**实质上是**<span style="background:#b1ffff">一个多接口的网桥</span>
	- 它能将网络分成小的冲突域，为每个用户提供更大的带宽。
	  
	- 对于传统**使用集线器**的共享式 10 Mb/s 以太网
		-  若共有 N 个用户,则每个用户的平均带宽为总带宽 (10 Mb/s) 的1/N
		  
	- 使用**以太网交换机** (<span style="background:#b1ffff">全双工方式</span>) 连接这些主机时，
		- 虽然从每个接口到主机的带宽还是 10 Mb/s，
		- 但是因为**一个用户通信时是独占带宽**的
			-  (而**不是和其他网络用户共享**传输介质带宽的), 
		- 所以拥有 N 个接口的交换机的总容量为 Nx 10 Mb/s。这正是交换机的**最大优点**
### 二、交换机的特点
- 交换机接口**直接与主机或者其他交换机连接**不使用CSMA/CD 协议
	- 当交换机的接口直接与主机或其他交换机连接时，
	- 可工作在全双工方式，并能同时连通多对接口，
	- 使每对相互通信的主机都能像独占通信介质那样，无冲突地传输数据，
	- 这样就不需要使用 CSMA/CD 协议。
	  
- 当交换机的接口**连接集线器**时，**只能使用 CSMA/CD 协议**且**只能工作在半双工方式**。
	- 当前的交换机和计算机中的网卡都能自动识别上述两种情况。
	  
- 交换机是一种**即插即用**设备，
	- 其内部的**帧转发表**是**通过自学习算法**，
	- 基于网络中各主机间的通信，<span style="background:#fff88f">自动地逐渐建立</span>的。
	  
- 交换机因为使用专用交换结构芯片，**交换速率较高**。
  
- 交换机**独占传输介质的带宽**。

### 三、以太网交换机的两种交换方式
#### 1. 直通交换方式
- <span style="background:#fff88f">只检查帧的目的 MAC 地址，以决定该帧的转发接口</span>。
  
- 这种方式的**交换时延非常小**，
- 缺点是它**不检查差错就直接转发**，因此可能**将一些无效帧转发给其他站**。
  
- 直通交换方式**不适用**于<span style="background:#d3f8b6">需要速率匹配、协议转换或差错检测</span>的线路。
#### 2. 存储转发交换方式
- <span style="background:#fff88f">先将接收到的帧缓存到高速缓存器</span>中，<span style="background:#fff88f">并检查数据是否正确</span>，<span style="background:#fff88f">确认无误后通过<font color="#ff0000">查找表</font>转换为输出接口</span>，以便将该帧发送出去。
- 若发现有错，则将其丢弃。
  
- 优点是**可靠性高**，且能**支持不同速率接口间的转换**
- 缺点是**时延较大**。
#### 3. 交换机的接口
- 交换机一般都具有多种速率的接口，如 10 Mbs、100 Mbs 的接口，以及多速率自适应接口。
### 四、交换机的自学习能力
#### 1. 基本概念
##### 1. 过滤
- 决定一个帧是转发到某个接口还是丢弃它——称为过滤
##### 2. 转发
- 决定一个帧应被移至哪个接口——称为转发
##### 3. 交换表
- 交换机的过滤和转发借助交换表 (switch table) 完成。
  
- 交换表中的一个表项至少包含
	- 一个 MAC 地址
	-  连通该 MAC 地址的接口
	  
![[Pasted image 20240426195938.png|425]]

#### 2. 交换机自学习的过程
![[Pasted image 20240426195949.png|400]]
- 例如，在图 3.35 中，
	- 以太网交换机有 4 个接口，各连接一台计算机，
	- MAC 地址分别为 A、B、C 和 D，
	- 交换机的交换表最初为空。

- A 先向 B 发送一帧，从接口 1 进入交换机。
	- 交换机收到帧后，查找交换表，找不到 MAC 地址为 B 的表项。
	- 然后，交换机将该帧的源地址 A 和接口 1 写入交换表，并向除接口 1 外的所有接口广播这个帧 
		- (该**帧就是从接口 1 进入的**，因此**不应将它再从接口 1 转发出去**)。
	- C 和 D 丢弃该帧，因为目的地址不对。
	- 只有 B 才收下这个目的地址正确的帧。
	- 交换表中写入 (A.1) 后，**从任何接口收到目的地址为 A 的帧都应从接口 1 转发出去**。
	- 这是因为，既然 A 发出的帧从接口 1 进入交换机，那么从接口 1 转发出去的帧也应能到达 A。

- 接下来，假定 B 通过接口 3 向 A 发送一帧，
	- 交换机查找交换表后，发现有表项 (A，1)，将该帧从接口 1 转发给 A。
	- 显然，**此时已没有必要再广播收到的帧。**
	- 将该帧的源地址 B 和接口 3 写入交换表，表明以后若有发送给 B 的帧，则应从接口 3 转发出去。

- 经过一段时间后，只要 C 和 D 也向其他主机发送帧，
	- 交换机就把 C 和 D 及对应的接口号写入交换表。
	- 这样，<span style="background:#fff88f">转发给任何主机的帧，就都能很快地在交换表中找到相应的转发接口</span>。
##### 注意
> [!note] 注意
> - 因为交换机所连的主机会随时变化，所以**需要更新交换表中的表项**。
> - 为此，<span style="background:#fff88f">交换表中的每个表项都设有一定的有效时间，过期表项将被自动删除</span>。
> - 这就保证了交换表中的数据符合当前网络的实际状况。
> - 这种自学习算法使得<span style="background:#fff88f">交换机能即插即用，而不必手工配置，因此非常方便</span>。

### 五、共享式以太网和交换式以太网的比较
**假设交换机已通过自学习算法逐步建立了完整的转发表，**
下面举例说明————>用**集线器**的共享式以太网与全部使用**交换机**的交换式以太网的区别。

|          |     | 集线器的共享式以太网                                                     |                             | 交换机的交换式以太网                                                             |
| -------- | --- | -------------------------------------------------------------- | --------------------------- | ---------------------------------------------------------------------- |
| 主机发送普通帧  |     | 集线器将帧转发到**其他所有接口**<br><br>其他各主机中的网卡根据帧的目的 MAC 地址决定接收或丢弃该帧。     |                             | 交换机收到帧后<br>根据帧的目的 MAC 地址和自身的交换表将帧**明确地转发给目的主机。**                       |
| 主机发送广播帧  |     | 集线器将帧转发到**其他所有接口，**<br><br>其他各主机中的网卡检测到帧的目的 MAC 地址是广播地址时，就接收该帧 | 效果上看是相同的<br><br>但它们的原理并不相同。 | 交换机检测到帧的目的 MAC 地址是广播地址<br><br>于是从其他所有接口转发该帧<br><br>其他各主机收到该广播帧后，就接收该帧。 |
| 多对主机同时通信 |     | 当多对主机同时通信时<br><br>**必然产生冲突**                                   |                             | 交换机能实现多对接口的高速并行交换<br><br>因此**不会产生冲突**。                                 |

|     | 广播域 |     | 冲突域 |
| --- | --- | --- | --- |
| 集线器 | 不隔离 |     | 不隔离 |
| 交换机 | 不隔离 |     | 隔离  |
